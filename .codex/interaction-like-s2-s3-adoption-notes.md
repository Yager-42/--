## 0. 先把话说清楚（避免你又开始理想化）

两份方案解决的是同一个问题：点赞 = **事实（有没有点）** + **派生（多少赞）**。

真正差异在于：用什么组件承载“高并发写入”和“热点读”，以及它们的成本。

结论（给 12 岁也能懂）：
- **能借鉴的：理念**（怎么消灭特殊情况、怎么对付热点、怎么定义成功语义）
- **不该硬搬的：指定组件栈**（TaiShan KV / TiCDC / 本地 WAL 这些，现阶段只会把你拖死）

## 1. 可借鉴点（每条都给：借鉴原因 + 不借鉴/风险原因）

> 说明：有些点我们已经在当前方案里用了，这里仍然写出来，避免实现者走回头路。

| 借鉴点 | 原文严格版在讲什么 | 当前方案怎么落 | 借鉴原因（为什么好） | 不借鉴/风险原因（什么时候不适用） |
| --- | --- | --- | --- | --- |
| 1) 禁止 toggle，只做 set-state | 写接口只接收 `desiredState=0/1` | `action=ADD/REMOVE` 等价 `desiredState=1/0` | 幂等天然、特殊情况少；重试不会把计数算飞 | 如果你允许 toggle，后面所有“去重/重放/补偿”都会变成地狱 |
| 2) “同一用户窗口内以最后一次为准” | buffer 里 `userId -> desiredState` 覆盖 | Redis `ops hash(userId->state)` 覆盖 | 把疯狂连点变成“最后一次状态”，不需要一堆 if/else | 如果你按事件逐条加减，会被重复消息/重试打爆 |
| 3) 动态窗口（热点更快，冷门更慢） | 1s~10s 动态 flush | `window_ms`/delayMs 可按热度调整 | 热点不积压、冷门不浪费 IO；成本更可控 | 全量改成 1~10s 会增加 MQ/DB 压力；需要监控数据支撑 |
| 4) L1 本地缓存只给“热点 key”启用 | HotKey Detector 识别后才用 Caffeine | 当前文档已给“可选热点治理” | 好品味：把“热点”变成可识别的正常路径，减少 Redis 读压 | 引入 detector + 本地缓存会增加运维/一致性复杂度；读不成为瓶颈时别做 |
| 5) 明确“成功语义”不是“DB 已一致” | `accepted=true` 代表 WAL 接住 | 当前返回 `success=true` 应解释为 Redis 接住 | 防止产品/前端误解；也利于后端验收与排障 | 如果业务要求“点一下必须立刻写 DB”，那不是点赞系统，是银行系统（成本会暴涨） |
| 6) requestId/eventId（可选）用于追踪与对账 | 原文常见做法：事件有唯一标识 | 当前接口可选加 `requestId`（不破坏旧调用） | 排障利器；把“这个请求去哪了”说清楚 | 强制客户端生成会破坏兼容；所以必须是可选增强 |
| 7) 把“状态”和“计数”当两类数据 | LikeState(Exists) + LikeCount(聚合) | `interaction_reaction` + `interaction_reaction_count` | 这就是正确的数据结构；计数永远是派生值 | 如果你把计数当真相，迟早出现负数/错数而且没人知道 |

## 2. 不借鉴点（当前不做，但要写明为什么）

| 不借鉴点 | 原文严格版在讲什么 | 为什么不借鉴（当前阶段） | 什么时候才值得重新考虑 |
| --- | --- | --- | --- |
| 1) 本地 WAL + 内存聚合器（Aggregator） | 先内存 hold，再写本地 WAL 防宕机丢数据 | 工程复杂度暴涨（文件回放、断点续传、滚动、校验）；当前方案写入真相已落在 Redis，不需要再造一套 WAL | 你的 Redis 可靠性达不到要求、且写入 QPS 高到 Redis 成本扛不住，才考虑“WAL+聚合器” |
| 2) TaiShan KV 作为 L3 状态底座 | 海量数据用高性能 KV 做 Exists | 当前项目栈没有；引入等于“重做存储体系”，无法在短期验收 | 状态数据量大到 MySQL/Redis 明显撑不住、且你有成熟 KV 团队与预算 |
| 3) TiDB + TiCDC 用来同步缓存 | 用 CDC 做 DB->Redis 同步，避免业务双写 | 你现在的链路是 Redis->DB 的延迟对齐；改成 CDC 等于翻转整个一致性模型 | 当你把 DB 定为唯一真相且缓存必须严格跟随时（更重、更贵） |
| 4) “识别热点后直接绕过 Redis，全走 L1”作为硬规则 | 热点读直返本地 | 当前方案可以做，但不应该作为“热点默认强制”（常态仍建议 L1-first：热点优先走本地，miss 再回源 Redis）；否则缓存失效/抖动时体验会乱 | 作为“应急熔断档”：Redis 读延迟/错误率飙升时，对热点 key 临时启用，短 TTL，恢复后自动关闭 |
| 5) “前端乐观更新当兜底”写进后端契约 | 后端挂了前端也先变红 | 这是产品策略，不是后端工程必做；写进后端契约只会让实现者分心 | 你明确要“体验优先到允许假成功”，并且愿意承担投诉/对账成本时 |

## 2.1 关于“热点绕过 Redis”的正确用法（它不是默认，但绝不是没意义）

你把它理解成“备用轮胎”就行：平时不用，但一旦 Redis 读被打爆，它能救命。

关键点：这里说的 **“不做默认”** 指的是：
**热点 key 也不默认走“L1-only（请求路径完全绕过 Redis）”。**

热点场景更推荐的“常态默认”是 **L1-first**：热点优先走本地，miss 再回源 Redis。
L1-only 只应该作为“应急熔断档”，否则你是在主动制造用户可见的抖动（某些节点的点赞数突然跳）。

推荐落地（从保守到激进）：

1) **L1-first（推荐默认的热点模式）**  
   热点 key：先查 L1，miss 再查 Redis 并回填；非热点 key：直接查 Redis。  
   这就已经能把 Redis 的热点读 QPS 砍掉一大截。

2) **L1-only（极端热点/应急模式）**  
   只有当 Redis 已经成为瓶颈（延迟飙升/错误率上升/连接池打满）时，允许热点 key 的 `count` 在**请求路径不读 Redis**：直接返回 L1 的短 TTL 值（例如 1 秒）。
   如果 L1 暂时没有值，允许你做两种选择（二选一，别同时搞）：
   - **保守**：允许“仅一次”回源 Redis 预热后再进入 L1-only（现实里这已经够用了）
   - **更硬**：直接返回上一次值（stale）并异步预热（请求不阻塞）
   这就是你说的“绕过 Redis”，但它应该是“按条件触发的应急档”，不是常态。
