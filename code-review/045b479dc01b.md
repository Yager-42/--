- Title: `refactor(counter): migrate counts to redis-truth ports`

## Summary

这次重构目标很明确：把“计数读模型”统一收敛到 `IObjectCounterPort/IUserCounterPort`，同时把点赞在线链路改成 **Redis 原子写为准、DB 延迟收敛**，并把评论点赞的旁路副作用从 `InteractionService` 下沉到 `ReactionLikeService`。

整体方向合理，关键路径（`applyAtomic` + `syncTarget`）设计也自洽；但是当前实现里有一个会影响线上正确性的高风险点：**评论点赞变更事件没有设置幂等 `eventId`**，会破坏 inbox 去重语义，导致事件可能被丢弃或重复累计。

## Findings

### Must Fix

- **[ReactionLikeService] COMMENT 点赞事件缺少幂等键：`CommentLikeChangedEvent` 未设置 `eventId`，导致 inbox 去重失效，可能重复累加。**
  - 位置：`project/nexus/nexus-domain/src/main/java/cn/nexus/domain/social/service/ReactionLikeService.java`
  - 证据：
    - `publishCommentLikeChanged(...)` 只设置了 `rootCommentId/postId/delta/tsMs`，未设置 `eventId`。
    - `CommentLikeChangedConsumer.onMessage(...)` 严格校验 `event.getEventId()` 非空且 `inboxPort.save(eventId, ...)` 才继续；因此现在事件要么被直接丢弃，要么在重试/重复投递时无法幂等（取决于 BaseEvent 默认值是否被覆盖）。
  - 影响：
    - 丢弃：评论 `like_count`/热榜刷新可能不触发，表现为“评论点赞数不涨/热榜不动”。
    - 非幂等：重复投递会导致 `commentRepository.addLikeCount` + `objectCounterPort.increment` 重复累计，出现计数漂移。
  - 建议修复：
    - 用 `requestId` 生成确定性的 `eventId`（例如 `comment_like_changed:<rid>`），保证非空且可去重。
    - 同时明确 `tsMs` 的语义（事件产生时间即可）。

### Should Fix

- **[ReactionLikeService] `syncTarget()` 重复调度使用了固定默认窗口：对 COMMENT 目标可能不符合预期。**
  - 位置：`project/nexus/nexus-domain/src/main/java/cn/nexus/domain/social/service/ReactionLikeService.java`
  - 现状：首次 pending 用 `defaultWindowMs(target)`，但 `syncTarget()` 收敛后如果窗口又进新 ops，会用 `reactionCachePort.getWindowMs(target, DEFAULT_WINDOW_MS)`（固定 5 分钟默认）。
  - 风险：COMMENT 设置了 `COMMENT_WINDOW_MS=1000`，但续挂回到 5 分钟，可能导致评论落库延迟显著变长。
  - 建议：把 `DEFAULT_WINDOW_MS` 改为 `defaultWindowMs(target)`，保持一致。

- **[ReactionCachePort] `getCountFromRedis()` 命名/语义疑似不符：实际会回源 DB 并回填。**
  - 位置：`project/nexus/nexus-infrastructure/src/main/java/cn/nexus/infrastructure/adapter/social/port/ReactionCachePort.java`
  - 现状：`getCountFromRedis` 调用 `redisGetCntOrRebuild`，缺失时会查 DB 并写回。
  - 建议：要么改名为 `getCountOrRebuild`/`getCountFromCacheOrDb`，要么补充注释避免误用（尤其在 “fromRedis” 语义下会被认为不触 DB）。

- **[RelationService] `unfollow()` 在 NOT_FOLLOWING 分支同时 `evict(FOLLOWER)` 可能过度失效。**
  - 位置：`project/nexus/nexus-domain/src/main/java/cn/nexus/domain/social/service/RelationService.java`
  - 现状：未关注时也清理 source/target 双方 FOLLOWING+FOLLOWER 四个 key。
  - 影响：更容易触发回源重建；如果这是“幂等修复”可以接受，但建议评估是否只需要清理本次涉及的计数（例如 source FOLLOWING、target FOLLOWER）。

### Nice To Have

- **[ReactionLikeService] 旁路异常处理不一致：部分 `catch (Exception ignored)` 会吞掉信号。**
  - 例如 `postLikeCachePort.applyCreatorLikeDelta` / `commentRepository.getBrief`。
  - 建议：至少打 debug/warn（可采样），避免线上排障困难。

- **[Counter ports] `batchGetCount` 返回 Map key 为 `hashTag()` 字符串：易用性一般。**
  - 如果后续调用者常需要按 targetId 回填，考虑提供 helper（或在调用侧统一封装）。

- **[Tests] 集成测试只等待通知队列 consumer readiness，未显式等待评论计数队列 consumer readiness。**
  - 例如 `ReactionHttpRealIntegrationTest.ensureNotifyConsumersReady()` 只检查 `InteractionNotifyMqConfig.Q_INTERACTION_NOTIFY`。
  - 这不一定会导致 flaky（容器通常都会 start），但当 MQ 拓扑/消费者启动慢时，comment like 断言可能受影响；可考虑加一个 `ensureCommentConsumersReady()`（至少检查 `InteractionCommentMqConfig.Q_COMMENT_LIKE_CHANGED/Q_REPLY_COUNT_CHANGED`）。

## Questions / Clarifications

- `CommentLikeChangedEvent` 的 `eventId` 期望是什么？是否允许复用请求 `requestId` 作为幂等键？
- COMMENT 点赞走 Redis 原子写 + 延迟落库后，评论主表 `like_count` 的真值来源是“事件增量”还是“最终对齐快照”？当前链路看起来两者都有（`CommentLikeChangedConsumer` + `ObjectCounterPort.rebuild` 回源）。
