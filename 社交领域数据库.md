以下是针对大流量、高并发场景的物理模型设计（Physical Data Model）。为了适配大厂标准，我特别强调了 **分库分表键（Sharding Key）** 的选择、**索引优化**以及**冷热分离**策略。

---

### **核心设计原则 (Design Principles)**

1. **ID 策略**：全站主键统一采用 **Snowflake（雪花算法）** 生成的 64 位 `BIGINT`，确保全局唯一且有序，天然适配 B+ 树索引。
2. **字符集**：统一使用 `utf8mb4`，支持 Emoji 表情存储。
3. **软删除**：所有核心表包含 `is_deleted` 字段，杜绝物理删除。
4. **通用字段**：每张表默认包含 `create_time`, `update_time`。
5. **分库分表**：在单表数据量预计超过 500 万行时，需根据 Sharding Key 进行拆分。

### 1. 用户关系服务域 (User Relationship Domain)

**设计痛点**：海量关系链，查询场景多为“关注列表”和“粉丝列表”。
**策略**：采用“写扩散”或“索引表”冗余，通常需要两份数据（正向表和反向表）。

```sql
-- 用户基础表 (核心信息)
CREATE TABLE `user_base` (
  `user_id` BIGINT NOT NULL COMMENT '用户ID (Sharding Key)',
  `username` VARCHAR(64) NOT NULL DEFAULT '',
  `avatar_url` VARCHAR(255) DEFAULT '',
  `privacy_level` TINYINT DEFAULT 0 COMMENT '隐私等级: 0公开, 1需审批, 2私密',
  `risk_status` TINYINT DEFAULT 0 COMMENT '风控状态: 0正常, 1冻结, 2Shadowban',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  `update_time` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`user_id`)
) ENGINE=InnoDB COMMENT='用户基础信息表';

-- 用户关系表 (正向: 我关注了谁 / 我屏蔽了谁)
CREATE TABLE `user_relation` (
  `id` BIGINT NOT NULL,
  `source_id` BIGINT NOT NULL COMMENT '发起方ID (Sharding Key)',
  `target_id` BIGINT NOT NULL COMMENT '目标方ID',
  `relation_type` TINYINT NOT NULL COMMENT '类型: 1关注, 2好友, 3屏蔽',
  `status` TINYINT DEFAULT 1 COMMENT '状态: 0无效, 1正常, 2等待审批(Pending)',
  `group_id` BIGINT DEFAULT 0 COMMENT '所属分组ID',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_source_target` (`source_id`, `target_id`, `relation_type`),
  INDEX `idx_source_status` (`source_id`, `status`)
) ENGINE=InnoDB COMMENT='用户关系正向表';

-- 粉丝表 (反向: 谁关注了我) - *用于高性能查询粉丝列表*
CREATE TABLE `user_follower` (
  `id` BIGINT NOT NULL,
  `user_id` BIGINT NOT NULL COMMENT '被关注者ID (Sharding Key)',
  `follower_id` BIGINT NOT NULL COMMENT '粉丝ID',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_user_follower` (`user_id`, `follower_id`),
  INDEX `idx_user_time` (`user_id`, `create_time`)
) ENGINE=InnoDB COMMENT='用户粉丝反向表';

```

### 2. 内容发布服务域 (Content Publication Domain)

**设计痛点**：读多写少，需要支持版本控制和多媒体关联。
**策略**：主表轻量化，内容大字段单独存或通过 OSS 链接，版本历史单独分表。

```sql
-- 内容主表
CREATE TABLE `content_post` (
  `post_id` BIGINT NOT NULL COMMENT '内容ID (Sharding Key: user_id取模 或 post_id)',
  `user_id` BIGINT NOT NULL,
  `content_text` TEXT COMMENT '文本内容 (若超长建议存ES/HBase)',
  `media_type` TINYINT DEFAULT 0 COMMENT '类型: 0纯文, 1图文, 2视频',
  `media_info` JSON COMMENT '媒体资源简要信息(宽表设计，减少连表)',
  `location_info` JSON COMMENT '地理位置信息',
  `status` TINYINT DEFAULT 1 COMMENT '状态: 0草稿, 1审核中, 2已发布, 3审核拒绝, 4定时发布中',
  `visibility` TINYINT DEFAULT 0 COMMENT '可见性: 0公开, 1好友, 2仅自己',
  `version_num` INT DEFAULT 1 COMMENT '当前版本号',
  `is_edited` TINYINT DEFAULT 0 COMMENT '是否被编辑过',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`post_id`),
  INDEX `idx_user_time` (`user_id`, `create_time`)
) ENGINE=InnoDB COMMENT='内容发布主表';

-- 内容-业务类型映射表（用户发布时提交 postTypes；一帖多类型）
CREATE TABLE `content_post_type` (
  `post_id` BIGINT NOT NULL COMMENT '内容ID',
  `type` VARCHAR(64) NOT NULL COMMENT '业务类型/主题（用户发布时提交）',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`post_id`, `type`),
  INDEX `idx_type` (`type`)
) ENGINE=InnoDB COMMENT='内容-业务类型映射表（一对多）';

-- 内容编辑历史表 (Version Control)
CREATE TABLE `content_history` (
  `history_id` BIGINT NOT NULL,
  `post_id` BIGINT NOT NULL COMMENT '关联内容ID',
  `version_num` INT NOT NULL COMMENT '版本号',
  `snapshot_content` TEXT COMMENT '内容快照',
  `snapshot_media` JSON COMMENT '媒体快照',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`history_id`),
  INDEX `idx_post_ver` (`post_id`, `version_num`)
) ENGINE=InnoDB COMMENT='内容版本历史表';

-- 草稿箱表 (支持云同步)
CREATE TABLE `content_draft` (
  `draft_id` BIGINT NOT NULL,
  `user_id` BIGINT NOT NULL,
  `draft_content` LONGTEXT,
  `device_id` VARCHAR(64) COMMENT '最后编辑设备',
  `client_version` BIGINT COMMENT '客户端版本号(用于冲突解决)',
  `update_time` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`draft_id`),
  INDEX `idx_user` (`user_id`)
) ENGINE=InnoDB COMMENT='云端草稿箱';

-- 定时发布任务表
CREATE TABLE `content_schedule` (
  `task_id` BIGINT NOT NULL,
  `user_id` BIGINT NOT NULL,
  `content_data` LONGTEXT COMMENT '待发布内容的完整Payload',
  `schedule_time` DATETIME NOT NULL COMMENT '预定发布时间',
  `status` TINYINT DEFAULT 0 COMMENT '状态: 0待执行, 1执行中, 2已完成, 3失败',
  `retry_count` INT DEFAULT 0,
  PRIMARY KEY (`task_id`),
  INDEX `idx_time_status` (`schedule_time`, `status`) COMMENT '用于Worker扫描'
) ENGINE=InnoDB COMMENT='定时发布任务表';

```

### 3. 互动服务域 (Interaction Domain)

**设计痛点**：极高的并发写入（点赞/评论）。
**策略**：核心计数采用 Redis，数据库作为归档；评论树形结构若过深，数据库层采用邻接表或闭包表，业务层限制层级。

```sql
-- 评论表 (支持两级结构: Root -> Reply)
CREATE TABLE `interaction_comment` (
  `comment_id` BIGINT NOT NULL,
  `post_id` BIGINT NOT NULL COMMENT '关联帖子ID (Sharding Key)',
  `user_id` BIGINT NOT NULL COMMENT '评论者ID',
  `root_id` BIGINT DEFAULT 0 COMMENT '根评论ID (0表示一级评论)',
  `parent_id` BIGINT DEFAULT 0 COMMENT '父评论ID (回复谁)',
  `content` VARCHAR(1024) NOT NULL,
  `like_count` INT DEFAULT 0 COMMENT '点赞数 (定期回写)',
  `is_pinned` TINYINT DEFAULT 0 COMMENT '是否置顶',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`comment_id`),
  INDEX `idx_post_root` (`post_id`, `root_id`) COMMENT '查询帖子下评论流',
  INDEX `idx_root_parent` (`root_id`, `create_time`) COMMENT '查询楼中楼'
) ENGINE=InnoDB COMMENT='评论表';

-- 互动流水表 (点赞/收藏/分享) - *量大，建议按月分表*
CREATE TABLE `interaction_action` (
  `id` BIGINT NOT NULL,
  `target_id` BIGINT NOT NULL COMMENT '目标ID (Post/Comment)',
  `target_type` TINYINT NOT NULL COMMENT '类型: 1帖子, 2评论',
  `user_id` BIGINT NOT NULL,
  `action_type` TINYINT NOT NULL COMMENT '1点赞, 2不喜欢, 3收藏, 4分享',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_target_user_action` (`target_id`, `target_type`, `user_id`, `action_type`)
) ENGINE=InnoDB COMMENT='用户互动行为流水表';

-- 点赞真值表（最终状态：谁给哪个目标点过赞）
-- 说明：为配合“Redis 秒回 + 延迟 flush 最终一致”，这里用 status 覆盖式 upsert 最终状态，避免靠 delete 表达取消赞带来的语义混乱。
CREATE TABLE `likes` (
  `user_id` BIGINT NOT NULL,
  `target_type` VARCHAR(32) NOT NULL COMMENT 'POST/COMMENT（与接口入参保持一致，避免额外映射）',
  `target_id` BIGINT NOT NULL,
  `status` TINYINT NOT NULL COMMENT '1=liked, 0=unliked',
  `create_time` DATETIME NOT NULL,
  `update_time` DATETIME NOT NULL,
  PRIMARY KEY (`user_id`, `target_type`, `target_id`)
) ENGINE=InnoDB COMMENT='点赞明细真值表（最终一致）';

-- 点赞计数表（最终一致的聚合真值）
-- 说明：flush 写绝对值（来自 Redis 当前 count），天然幂等，重复 flush 不会漂移。
CREATE TABLE `like_counts` (
  `target_type` VARCHAR(32) NOT NULL COMMENT 'POST/COMMENT',
  `target_id` BIGINT NOT NULL,
  `like_count` BIGINT NOT NULL DEFAULT 0,
  `update_time` DATETIME NOT NULL,
  PRIMARY KEY (`target_type`, `target_id`)
) ENGINE=InnoDB COMMENT='点赞计数真值表（最终一致）';

-- 通知收件箱（最小 MVP：站内通知列表）
-- 说明：点赞通知由异步 consumer 入库；fingerprint 用于去重（重复消息/重复点赞不会刷屏）。
CREATE TABLE `notification_inbox` (
  `id` BIGINT NOT NULL,
  `to_user_id` BIGINT NOT NULL COMMENT '被通知的人',
  `from_user_id` BIGINT DEFAULT NULL COMMENT '触发者（点赞的人）',
  `event_type` VARCHAR(32) NOT NULL COMMENT 'LIKE/COMMENT/...（最小先做 LIKE）',
  `target_type` VARCHAR(32) DEFAULT NULL COMMENT 'POST/COMMENT',
  `target_id` BIGINT DEFAULT NULL,
  `title` VARCHAR(255) NOT NULL,
  `content` VARCHAR(1024) NOT NULL,
  `status` TINYINT NOT NULL DEFAULT 0 COMMENT '0未读, 1已读',
  `fingerprint` VARCHAR(128) NOT NULL COMMENT '幂等去重指纹',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  `update_time` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_fingerprint` (`fingerprint`),
  INDEX `idx_to_time` (`to_user_id`, `create_time`)
) ENGINE=InnoDB COMMENT='通知收件箱（站内通知列表）';

-- 投票主表
CREATE TABLE `interaction_poll` (
  `poll_id` BIGINT NOT NULL,
  `post_id` BIGINT NOT NULL,
  `title` VARCHAR(255),
  `allow_multi` TINYINT DEFAULT 0 COMMENT '多选',
  `expire_time` DATETIME,
  PRIMARY KEY (`poll_id`)
) ENGINE=InnoDB;

-- 投票选项与计数
CREATE TABLE `interaction_poll_option` (
  `option_id` BIGINT NOT NULL,
  `poll_id` BIGINT NOT NULL,
  `option_text` VARCHAR(255),
  `vote_count` INT DEFAULT 0 COMMENT '票数',
  PRIMARY KEY (`option_id`),
  INDEX `idx_poll` (`poll_id`)
) ENGINE=InnoDB;

```

### 4. 钱包与资产服务域 (Wallet & Tipping Domain)

**设计痛点**：资金安全，事务一致性。
**策略**：严格的事务控制，乐观锁，流水不可篡改。

```sql
-- 用户钱包账户
CREATE TABLE `wallet_account` (
  `user_id` BIGINT NOT NULL,
  `currency_type` VARCHAR(16) NOT NULL DEFAULT 'CNY',
  `balance` DECIMAL(18, 4) NOT NULL DEFAULT 0.0000,
  `frozen_amount` DECIMAL(18, 4) NOT NULL DEFAULT 0.0000 COMMENT '冻结金额',
  `version` BIGINT NOT NULL DEFAULT 0 COMMENT '乐观锁版本号',
  `update_time` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`user_id`, `currency_type`)
) ENGINE=InnoDB COMMENT='用户钱包表';

-- 交易流水表
CREATE TABLE `wallet_transaction` (
  `tx_id` BIGINT NOT NULL,
  `from_uid` BIGINT NOT NULL,
  `to_uid` BIGINT NOT NULL,
  `amount` DECIMAL(18, 4) NOT NULL,
  `biz_type` TINYINT NOT NULL COMMENT '业务: 101打赏, 201充值',
  `related_id` BIGINT COMMENT '关联业务ID (如 post_id)',
  `status` TINYINT DEFAULT 1 COMMENT '1处理中, 2成功, 3失败',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`tx_id`),
  INDEX `idx_from` (`from_uid`),
  INDEX `idx_to` (`to_uid`)
) ENGINE=InnoDB COMMENT='资金交易流水表';
```

### 架构师补充说明 (Architect's Notes)

1. **关于 Feed 流 (Timeline)**：
* **不要用 MySQL 存储 Feed 流数据**。上述设计中没有 `feed_timeline` 表，因为 Feed 流（收件箱）应该使用 **Redis (List/ZSet)** 或 **HBase/Cassandra** 这种 NoSQL 数据库来存储。
* MySQL 仅存储元数据（Post 表）和关系数据（Relation 表）。


2. **关于搜索 (Search)**：
* 搜索功能不直接查询 MySQL。MySQL 通过 **CDC (Canal/Debezium)** 将 `content_post` 的变更实时同步到 **Elasticsearch**。


3. **负反馈 (Negative Feedback)**：
* 负反馈数据（Users x Items）极其稀疏且量大，建议存入 **ClickHouse** 或 **Hive** 用于离线推荐模型训练，线上使用 Redis BloomFilter 做实时去重。
