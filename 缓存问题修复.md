# 缓存问题修复.md

适用方案：`缓存/nexus-redis-统一实施与审计合并方案-2026-03-09.md`  
本文目标：把“缓存问题.md”里列出的偏差，按统一方案对齐修复；不发明新协议，也不再为已有缓存内容额外复制一层 key。  

## 统一口径（写死，避免分歧）

### 核心协议定义（必须严格遵守）

**协议 1：`interact:content:post:*` 的真义**
- 定义：存储 `selectById` 查出来的”帖子真相对象”（所有状态，无过滤）
- 写入资格：只有走 `selectById` 真相源的路径（如 `findPost()`）才能写入此 key
- 读取后处理：`listPostsByIds()` 等需要过滤状态的场景，必须在读到缓存后再过滤 `status=2`，不能直接全收
- 禁止行为：不允许把”published-only 查询没查到”的对象写成 NULL（因为这不等于”不存在”）

**协议 2：NULL 哨兵的写入资格**
- 只有走 `selectById` 真相源的路径，确认 DB 真没有记录，才允许写 `”NULL”`
- `listPostsByIds()` 走 `selectByIds`（只查 status=2）的路径，对 miss 的 id **禁止写 NULL**
- 原因：`selectByIds` 的 miss 可能只是”不是已发布状态”，不代表帖子不存在

**协议 3：`FeedCardStatRepository` 的迁移策略**
- 不直接删除类和接口，先变成薄适配层
- `getBatch()` 改为转调 `ReactionCachePort.getCount()` 批量版本
- `saveBatch()` 改为空实现（因为 `ReactionCachePort` 的写入由点赞链路统一管理）
- 目的：避免 `SearchService` 等调用方在迁移半路断掉

**协议 4：详情页正文来源**
- 详情页应复用 `ContentRepository.findPost()` 返回的 `post.getContentText()`
- `findPost()` 已经调用 `fillPostContent()` 从 KV 读取并填充了 `contentText`
- 详情页不应该命中帖子缓存后又再查一次 KV
- 只有 `contentText` 为空时，才允许兜底查 KV

### 通用规则

- 输入不合法：直接报”参数错误”；不打缓存、不回源。
- 真相源回源失败（DB/KV/下游失败）：
  - HTTP：直接返回明确错误
  - MQ：直接抛错触发重试
  - 共同要求：不拼半成品，不做“看起来能用”的降级
- MQ 失败处理：先重试；重试仍失败必须落补偿（由补偿任务重跑），不允许静默吞掉。
- “落补偿”的默认实现：让消息进入 DLQ，由可靠消息重放机制记录失败并后续重放（例如 `ReliableMqDlqRecorder -> ReliableMqReplayService` 这类模式）。
- “业务链路”范围：包含用户实时接口（HTTP）和线上常跑的异步链路（MQ 消费、建索引、通知等）；只要会在生产里持续打 DB，都算。
- “经常直打 DB”的判断口径：只要它出现在业务链路的主流程循环里（例如一页 20 条，每条都查一次 DB），就算“经常”；否则按监控证据再决定。
- 公共缓存口径：跨请求缓存里只允许放“所有用户看到都一样”的公共稳定数据；权限、风控、viewer 态、用户态一律不进公共缓存。
- 指标口径：只有“本身就先在缓存里完成实时计算，再异步更新 MQ/下游”的指标，才允许继续放在现有指标 key；业务组装层只能读这个现有 key，不允许再派生第二个业务镜像 key。
- 复用口径：同一份公共数据只允许一套跨请求 key 协议；如果现有缓存已经能找到对应内容，就直接复用，不允许再新建 `detail/stat/summary` 之类 key 再包一层。
- 性能阈值口径：不写具体毫秒阈值；只验收三条形状规则：
  - 命中某块缓存（L1/L2）时，该块不得再回源它对应的真相源（不得打它那张表/那条查询）
  - 不允许把“一页 N 条同类数据”变成“DB 查询 N 次”的固定模式
  - 不允许为了省一次组装，再把已有缓存里的内容复制成新的跨请求 key
- 用户态字段口径：`liked/followed/seen/isBlocked` 等一律每次实时查/算，不做任何轻缓存；且永远不进公共缓存。

## 目标用户/场景（泛化描述）

- 普通用户：刷首页/热门等 Feed 列表；看个人主页列表；点进帖子详情；看评论与回复预览。
- 业务高峰：活动、热点内容出现时，读请求暴增但页面还能打开，不会因为缓存抖动把 DB 打爆，也不会因为重复 key 导致失效链路失控。
- 研发排障：当出现“突然打库变多/页面变慢”，能按统一规则快速定位是缓存 miss、失效不完整、还是有人又加了一层重复缓存。
- 系统作业：MQ 消费等异步任务高频运行时，不会把 DB 打爆，也不会因为缓存分裂导致不可预测的回源放大。

## MVP 能力边界（做什么）

- 只做“对齐统一方案写死的执行契约”：key、TTL（含抖动）、`NULL` 哨兵、single-flight、写后失效、降级语义。
- 优先修复业务链路主流程循环里的直打 DB 点，优先复用已有公共缓存与已有指标缓存，不靠新增镜像 key 兜底。
- 修复会制造永久脏 key 的写入缺陷：写缓存必须一次性带 TTL，避免 `SET` 后再 `EXPIRE` 的分离写法。
- 修复“同一个对象两套协议”的分裂：同一种公共对象读取要走同一套 L1/L2/回源规则，不允许旁路绕过，也不允许复制成第二套跨请求 key。
- Redis 故障要能用：Redis 超时/宕机/JSON 解析失败都当 miss；回源成功就回填；缓存异常不能让业务失败。
- 指标类数据只保留现有“缓存先算、再异步 MQ”的那一层；Feed/详情等业务层只读取它，不再复制统计快照。

## 明确不做什么（防膨胀）

- 不新造缓存框架，不引入新的大而全 `CacheService`。
- 不把 `liked/followed/seen/isBlocked` 等“跟谁在看有关”的字段塞进公共缓存。
- 不重启关系邻接缓存协议：不引入 rebuild、`tmpKey`、分布式锁、`rename` 切换、镜像补写等复杂机制。
- 不新增 `feed:card:stat:*`、`interact:content:detail:*` 这类“已有缓存里能拼出来，却再存一份”的跨请求镜像 key。
- 不对白名单外的 key 开“热点 TTL 自增长”；尤其不对 `interact:reaction:*`、`comment:*` 做热延寿。
- 没有真实监控证据（QPS/回源耗时/热点分布）的点，先不做“额外缓存覆盖/跨实例协调/更激进的优化”。

## 关键边界情况（输入归一化/异常/权限/失败兜底）

### 输入归一化

- id 为空：直接报参数错误（不打缓存、不回源）。
- 批量 id：去重（避免重复回源），同时保持原顺序（保证用户看到的顺序不变）。
- limit：做上限保护；例如回复预览只允许 `1..10`；超出范围直接报参数错误（不打缓存、不回源）。

### 异常与语义

- “不存在”和“失败”必须分开：只有真相源明确不存在，才写 `NULL`；Redis 超时/宕机/解析失败只算失败，不能写 `NULL`。
- JSON 解析失败：先删坏 key，再按 miss 回源（不允许把坏值继续留着）。
- 指标写路径（仅限本来就靠缓存先算的指标）：先在现有指标缓存里完成原子更新，再发 MQ；MQ 失败仍按重试/DLQ 补偿处理，不允许因为失败再新增一层统计 key 兜底。

### 权限与用户态

- 公共缓存只放稳定字段；权限、风控、是否点赞/关注/已读等一律每次按当前用户实时计算/查询；不做轻缓存；且不进公共缓存。
- 任何“跟查看者有关”的字段，永远最后补，永远不参与公共缓存命中。
- 详情页和 Feed 卡片如果需要作者资料、互动数、viewer 态，统一走“现有公共缓存/现有指标缓存 + 最后补用户态”的组装顺序，不再落第二份快照 key。

### 失败兜底

- Redis 挂了：按 miss 回源 DB/KV/下游；回源成功再回填缓存。
- 真相源回源失败（DB/KV/下游失败）：
  - HTTP：直接返回明确错误
  - MQ：直接抛错触发重试；重试仍失败落补偿任务重跑
  - 不拼半成品，不做降级

### 并发与去重

- 同一热点 miss：单机内必须去重（single-flight），避免同一份数据被重复回源。
- 不做跨实例锁：允许“多台机器各回源一次”的上限，但不能因为分裂协议或重复 key 导致不可控放大。

## 验收标准（Checklist）

- [ ] 文档与实现一致：涉及的 key/TTL/`NULL`/single-flight/失效/降级语义与统一方案一致。
- [ ] Feed 主链路：拉一页 Feed 时，`likeCount` 不允许出现“每条都直查 DB”的固定模式；命中现有 `interact:reaction:cnt:*` 时，不允许查询 `interaction_reaction_count`。
- [ ] 无重复 key：如果 `interact:content:post:*` 或 `interact:reaction:cnt:*` 已经能提供数据，就不会再写 `interact:content:detail:*`、`feed:card:stat:*` 这类镜像 key。
- [ ] 原子 TTL：写入 `feed:card:*` 等 key 不会出现“写成功但没 TTL”的永久脏 key。
- [ ] 语义正确：Redis 解析失败会删坏 key 并回源；不会把基础设施故障当“不存在”写 `NULL`。
- [ ] 个性化不串：`liked/followed/seen` 不进入公共缓存；不同用户不会互相污染。
- [ ] 失效有效：写后失效会同时清理 Redis 与本地 L1（如有）；延迟双删不会因为排队变成“延迟 N 秒”。
- [ ] 故障可用：Redis 故障时，只要真相源可用，主流程仍能回源返回正确结果；缓存故障不会导致业务失败。
- [ ] MQ 链路：真相源回源失败会触发重试；重试仍失败会进入补偿重跑路径；不允许静默吞掉。

## 工程实现计划

### 模块边界

- `nexus-domain`：只放接口和 VO（例如 `IContentRepository`、`IFeedCardRepository`、`FeedCardVO`）；不写 Redis、不写缓存策略。
- `nexus-infrastructure`：实现 Redis/L1/single-flight/热点 TTL 延寿等“读路径内部缓存”；主要改动点：`ContentRepository`、`FeedCardRepository`、`ReactionCachePort` 以及相关组装辅助类。
- `FeedCardStatRepository` 若还保留：只能作为“从现有指标缓存取数”的薄适配层，不能再拥有自己的 Redis key 协议。
- `nexus-trigger`：实现失效广播与本地 L1 清理、MQ 重试与 DLQ 记录；主要改动点：`ContentCacheEvictPort`、`ContentCacheEvictConsumer`，以及需要遵守“统一口径”的 HTTP/MQ 调用方。
- 可靠补偿：复用现有 DLQ + 重放机制作为“补偿任务重跑”的落地点（不新建系统）。

### 数据结构

- Redis key（按统一方案）：
- `interact:content:post:{postId}`：value 为稳定 JSON 或 `"NULL"`；TTL `60~75s`；热点时保留现有 `isHotKey -> 固定 TTL 上限` 延寿策略（只对白名单正向值生效）。
- `feed:card:{postId}`：value 只包含稳定公共字段；TTL `30~40m`；热点延寿只作用于此 key 家族，并保留现有 `isHotKey -> 固定 TTL 上限` 策略。
- `interact:reaction:cnt:{tag}`：现有指标真相前置层；先在这里原子更新，再异步走 MQ/下游；不设置 TTL；丢失时回表 `interaction_reaction_count` 重建。
- 明确禁止新增：
- `feed:card:stat:{postId}`：`likeCount` 等统计禁止再复制一层
- `interact:content:detail:{postId}`：详情公共数据禁止再落一份跨请求镜像 key
- 本地 L1：
- 热点 L1：`Caffeine`，TTL 固定 `2s`，只用于允许的热点 key 家族。
- single-flight：
- 复用现有 `SingleFlight`（本机并发去重）；不做跨实例锁。
- 热点 TTL 延寿（本次保留现状）：
- 继续使用现有 `isHotKey` 判定热点；只对白名单正向公共值做固定 TTL 上限延寿；本次只去掉读路径延寿时“GET value 再 EXPIRE”的额外往返，不改成 `60s` 滑窗、`6 bucket`、分级上限算法。

### 构建方式

- 代码在 `project/nexus`（多模块 Maven）。
- 建议本地构建顺序：
- 先跑单测：`mvn -pl nexus-infrastructure,nexus-trigger,nexus-domain -am test`
- 再打包：`mvn -pl nexus-app -am package`
- 若需要跳过耗时用例：只允许临时 `-DskipTests` 用于本地验证，合入前必须补回测试。

### 测试策略

- 单元测试（优先，跑得快，稳定）：
- `FeedCardAssemblerOrReactionCacheTest`：同一批 `postIds` 并发 miss 时，统计读取只通过现有 `interact:reaction:cnt:*` 重建 1 次；命中现有指标缓存时不查 `interaction_reaction_count`；不会写 `feed:card:stat:*`。
- `ContentRepositoryFindPostTest`：`findPost` 不允许旁路绕过 L2 + single-flight；命中 `"NULL"` 不打 DB；并发 miss 只回源 1 次。
- `ContentDetailQueryServiceTest`：详情读取复用 `interact:content:post:*`、正文/KV、作者缓存、指标缓存组装；不新增跨请求 detail key。
- `ContentCacheEvictPortTest`：`evictPost` 必须同时清理本地 L1 + Redis，并发出 MQ 广播；延时双删不会因为排队变成“延迟 N 秒”。
- 集成测试（可选，有环境再做）：
- 连接真实 Redis，验证写入 key 的 TTL 符合区间；模拟 Redis 宕机，验证按 miss 回源；验证不会生成 `feed:card:stat:*` 这类重复 key。
- 人工验收（必须，贴近真实链路）：
- 拉一页 Feed（例如 20 条）时，不允许出现“每条都查询 `interaction_reaction_count`”的固定模式。
- 帖子详情与 Feed 卡片命中已有缓存后，不会再创建一份重复公共快照 key。
- Redis 故障时，只要真相源可用，接口能回源返回正确结果。
- MQ 消费失败会自动重试；重试仍失败会进入 DLQ 并被记录，后续可重放。

### 风险与降级

- 风险：严格口径会让“真相源抖动”直接变成 HTTP 报错或 MQ 重试，短期可用性会下降，但实现口径更一致。
- 风险：不再复制统计或详情镜像 key 后，部分链路会多一次组装，但能显著降低 key 分裂、失效遗漏和脏数据窗口。
- 风险：热点 TTL 延寿若实现不当会放大陈旧窗口；控制手段是白名单 + 只在已拿到正向对象时延寿 + 写路径统一 `evictPost`。
- 降级策略（写死）：
- Redis 故障：按 miss 回源并回填；不把 Redis 故障翻译成“不存在”。
- 真相源失败：HTTP 直接报错；MQ 抛错触发重试，最终进入 DLQ + 补偿重放；不拼半成品。
