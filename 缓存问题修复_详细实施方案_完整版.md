## F1. 删除 `feed:card:stat:*` 二层缓存口径

> **⚠️ 此任务已重写，详见 `缓存问题修复_重写版_F1.md`**
>
> 重写原因：原版循环调用 `reactionCachePort.getCount()` 在冷 miss 时仍退化成 N 次查库；
> 且未覆盖 `SearchService` 的迁移策略，删除 `FeedCardStatRepository` 会导致编译断链。

### 核心变化摘要

- `IReactionCachePort` 新增 `batchGetCount()` 批量接口
- `ReactionCachePort` 实现批量接口：批量 `multiGet` Redis，miss 的批量回表
- `FeedCardAssembleService` 改为调用 `batchGetCount()`，不再循环单条查
- `FeedCardStatRepository` 变为薄适配层，`getBatch()` 转调 `batchGetCount()`，`saveBatch()` 空实现
- `SearchService` 调用方不变（仍调 `FeedCardStatRepository`），薄适配层保证不断链
- `ContentCacheEvictPort` 和 `ContentCacheEvictConsumer` 删除对 `FeedCardStatRepository` 的依赖

### 全量调用方清单

| `SearchService.java` 第 30/136/159 行 | 不改，薄适配层保证不断链 |

### 验收标准

- 冷启动拉 20 条 Feed：`interaction_reaction_count` 只被查询 1 次（批量），不是 20 次
- Redis 中没有 `feed:card:stat:*` key

## F5. `ContentDetailQueryService` 跨节点 miss 协调

### 依赖
- S2 (决策确认) ✅

### 验收标准
- `rebuildSnapshot` 内 singleFlight leader 回源前必须先二次查 Redis
- 多节点同时 miss 同一热点详情时，第一个节点写回 Redis 后，其余节点的 singleFlight 结束后能从 Redis 命中，不再各自打 DB+KV
- 不引入分布式锁，不引入跨实例 singleFlight

### 问题分析

当前 `rebuildSnapshot`（第 188-236 行）的流程：
```
singleFlight.execute(postId, () -> {
    二次查 Redis  ← 已有，但只在 singleFlight 内部
    回源 DB+KV
    写回 Redis
})
```

问题在于：`singleFlight` 是进程内的 `SingleFlight`，只能保证**同一节点**的并发 miss 只回源一次。多节点场景下，节点 A 的 singleFlight leader 正在回源时，节点 B 也在独立回源，两者都会打 DB+KV。

**解法**：不需要分布式锁。节点 B 的 singleFlight leader 在回源前先二次查 Redis——如果节点 A 已经写回，节点 B 直接命中返回，不再打 DB+KV。这与 `ContentRepository.listPostsByIds` 的 leader 内二次查 Redis 逻辑完全一致。

当前代码第 188-210 行的 `rebuildSnapshot` 已经有二次查 Redis 的逻辑，但**位置不对**：它在 `singleFlight.execute` 的 lambda 内部，而 lambda 本身就是 singleFlight 的 supplier，这意味着二次查 Redis 是在 leader 持有锁之后才执行的，逻辑上是正确的。

仔细核对代码后，`rebuildSnapshot` 第 190-204 行确实在 leader 内做了二次查 Redis，流程是：
1. leader 内二次查 Redis（第 190 行）
2. 命中则直接返回（第 200 行）
3. miss 才回源 DB+KV（第 212 行）

**真正的问题**是：`query` 方法第 94 行调用 `singleFlight.execute` 后，**非 leader 的等待者**拿到 leader 的结果直接返回，不会再查 Redis。但 leader 已经把结果写回 Redis 了，所以等待者拿到的是正确结果，这部分没问题。

**实际缺口**：多节点场景下，节点 B 在节点 A 的 singleFlight 执行期间发起请求，节点 B 走到 `singleFlight.execute` 时，节点 B 自己的 singleFlight 是独立的（进程内），节点 B 的 leader 会独立回源。**修复方式**：节点 B 的 leader 在 `rebuildSnapshot` 开头的二次查 Redis 能命中节点 A 写回的值，就不再回源。这个逻辑**已经存在**于第 190-204 行。

**结论**：`rebuildSnapshot` 的二次查 Redis 逻辑已经正确实现了跨节点协调。但需要确认一个细节：`query` 方法第 66-69 行先查本地 L1，第 71-92 行查 Redis L2，第 94 行才进 singleFlight。如果节点 B 在节点 A 写回 Redis 之前就已经过了第 92 行（Redis miss），进入 singleFlight，那么节点 B 的 leader 在 `rebuildSnapshot` 第 190 行会再次查 Redis，此时节点 A 已写回，节点 B 命中，不再回源。**这个路径是正确的**。

**真正需要修复的**是：`query` 方法第 94 行 `singleFlight.execute` 的返回值直接写入 `localCache`（第 95 行），但如果 singleFlight 内部抛出 `AppException`（NOT_FOUND），会穿透到调用方，不会写 localCache，这是正确的。但如果 singleFlight 内部抛出其他异常，当前代码没有处理，会直接抛给调用方，localCache 不会写入脏数据，也是正确的。

**最终结论**：`ContentDetailQueryService` 的跨节点协调通过 Redis L2 已经实现，不需要额外修改。但需要在文档中明确说明这一点，避免后续误判。
