# 查漏.txt → 修复执行指南（按当前项目实现落地）

目标：把 `查漏.txt` 里“已经明确的修改点”变成可直接照着做的步骤；把“提问/疑点”先定位到代码里的具体点，再给出明确改法。  
范围：默认只改 `project/nexus/**` 这套 Spring Boot 工程。

> 你已确认的额外规则（本指南已按此更新）：
> 1) **draftId = postId**：禁止 `publish(postId=null)` 新建 postId。
> 2) **设置定时发布后：不能再编辑草稿或 post 内容**；想编辑必须先取消/删除该定时任务。
> 3) 点赞去重必须用 **Redis bitmap**（不再用每个 target 一个 Redis set）。
> 4) “点赞延时落库”的指数退避重试：必须用 **Redis ZSET + Lua** 的实现方案。

## 0. 先对齐一个总原则：先把“ID/版本”收口，后面很多问题会一起消失

你在 `查漏.txt` 里提到的这几类问题，本质都指向同一件事：

- **没有稳定的 postId** → 锁、幂等、定时发布重试都会失效
- **异步回写只看状态** → 容易把“旧审核结果”套到“新内容版本”
- **删除/发布并发规则不统一** → 会出现“删了又活/活了又删”

所以本指南的第一条“总改法”是：

### 0.1 统一：草稿 draftId = postId（强约定）

做法（必须按这个做，后面步骤都基于此）：

1. **新发帖必须先拿一个 ID**：客户端先调 `PUT /api/v1/content/draft` 拿到 `draftId`。
2. **后续把这个 `draftId` 当成 `postId` 用**：发布、定时发布、草稿同步都用同一个 ID。
3. **服务端禁止“publish(postId=null) 新建 postId”**：否则并发下必然生成两条帖子。

这样做会带来的直接收益：

- “锁”的粒度固定（按 postId），并发不会分裂成两条帖子
- 幂等 token 不会因为 postId 每次变而失效
- 定时发布重试不会变成“多发 N 条”

### 1.2 「草稿正文/媒体不可以为空（保持简单）」

**现状定位**

- 草稿保存：`project/nexus/nexus-domain/src/main/java/cn/nexus/domain/social/service/ContentService.java` → `saveDraft()`  
  目前允许 `contentText=null` 且 `mediaIds=null`
- 草稿同步：`ContentService.syncDraft()`  
  目前允许 `diffContent` 为空；也不校验媒体
- 发布：`ContentService.publish()`  
  目前允许 `text` 和 `mediaInfo` 都空
- 定时发布：`ContentService.schedule()`  
  目前允许 `contentData` 为空

**要改成什么（唯一规则，别加花）**

- “内容”满足其一即可：
  - `text/草稿正文` **非空且非全空格**；或
  - `mediaIds/mediaInfo` **至少有 1 个**
- 两者都没有 → 直接返回 `ResponseCode.ILLEGAL_PARAMETER`（`0002`）

**具体步骤**

1. 新增一个共用校验函数（放在 `ContentService` 私有方法）：
   - 入参：`String text, List<String> mediaIds, String mediaInfo`
   - 规则：`text` trim 后有内容 **或** `mediaIds` 非空 **或** `mediaInfo` trim 后有内容
2. 在下面 4 个入口调用校验（必须全加）：
   - `ContentService.saveDraft()`
   - `ContentService.syncDraft()`
   - `ContentService.publish()`
   - `ContentService.schedule()`
3. 校验失败时统一抛 `AppException(ResponseCode.ILLEGAL_PARAMETER, "content 不能为空")`

**验收**

- 传空草稿保存：返回 `0002`
- 传空发布：返回 `0002`
- 传只有媒体：成功
- 传只有文本：成功

### 1.3 「每个 post 的草稿应该只能有一个」

**现状定位**

- 当前 `content_draft` 只有 `draft_id` 主键，没有 `post_id` 字段：`project/nexus/docs/social_content_tables.sql`
- 当前 `PUT /api/v1/content/draft` 每次都会生成新 `draftId`：`ContentService.saveDraft()`

**要改成什么（与 0.1 一致）**

- **一个 postId 只有一条草稿**：草稿的主键就是 postId（即 draftId=postId）。
- “再次保存草稿”必须是覆盖更新同一条草稿（而不是新建一条）。

**具体步骤**

1. 改 API：让“保存草稿”支持覆盖更新（保持兼容）  
   - 修改 `SaveDraftRequestDTO`：新增字段 `Long draftId`（可空）
   - 规则：
     - `draftId` 为空：创建新草稿，生成新 ID（也是未来 postId）
     - `draftId` 不为空：覆盖更新该草稿（必须校验归属）
2. 改实现：`ContentService.saveDraft(userId, contentText, mediaIds)` 改为：
   - 入参增加 `draftId`
   - 若 `draftId==null`：`draftId = socialIdPort.nextId()`
   - `contentRepository.findDraft(draftId)`：
     - 若存在且 `draft.userId != userId`：拒绝（NO_PERMISSION）
   - `contentRepository.saveDraft(entity)` 走 `insertOrUpdate`
3. 同步改 Controller：`ContentController.saveDraft()` 传入 `requestDTO.getDraftId()`。

**验收**

- 同一 userId 用同一个 `draftId` 连续保存 10 次 → DB 里只有 1 行草稿
- 不同 userId 传同一个 `draftId` 覆盖 → 被拒绝

### 1.4 「PATCH /content/draft/{draftId} 的登录/归属校验」

**现状定位**

- 登录：全局由拦截器强制（必须带 `X-User-Id`）  
  - `project/nexus/nexus-trigger/src/main/java/cn/nexus/trigger/http/support/UserContextInterceptor.java`

- 归属校验：`ContentService.syncDraft()` **没有校验 userId**
  - `ContentController.syncDraft()` 也没有取 `UserContext.requireUserId()`

**要改成什么**

- 登录：保持现状（拦截器做）
- 归属：必须在业务层做（因为要查草稿归属，不能把 DB 访问塞进 interceptor）

**具体步骤**

1. 改 Controller：`ContentController.syncDraft()` 开头加：
   - `Long userId = UserContext.requireUserId();`
2. 改接口：`IContentService.syncDraft(...)` 增加入参 `Long userId`
3. 改实现：`ContentService.syncDraft(...)`：

   - `entity==null`：返回 NOT_FOUND（**不要**自动创建，避免被人撞库创建垃圾草稿）
   - `entity.userId != userId`：返回/抛 NO_PERMISSION
4. 只允许 owner 更新：更新 draftContent/mediaIds/deviceId/clientVersion/updateTime

**验收**

- 未登录（不带 header）→ 被拦截器挡下（`0002`）
- 登录但用别人的 draftId → NO_PERMISSION
- 登录且是自己的 → 成功

### 1.5 「clientVersion 必须是数字」

**现状定位**

- DTO：`project/nexus/nexus-api/src/main/java/cn/nexus/api/social/content/dto/DraftSyncRequestDTO.java` 的 `clientVersion` 是 `String`
- DB：`content_draft.client_version` 是 `BIGINT`
- 版本比较：`ContentService.isNewerClientVersion()` 解析失败会 `return true`（等于放行）

**要改成什么（不要再接收字符串）**

- `clientVersion` 改成 `Long`
- 版本比较规则：`incoming >= current` 才允许覆盖
- incoming 为空 → 视为非法参数（拒绝）

**具体步骤**

1. 改 DTO 类型：
   - `DraftSyncRequestDTO.clientVersion: String -> Long`
2. 改领域实体与 PO：
   - `ContentDraftEntity.clientVersion: String -> Long`
   - `ContentDraftPO.clientVersion: String -> Long`
3. 改 mapper/转换：
   - `ContentRepository.toDraftPO/toDraftEntity` 对应字段改成 Long
4. 改版本比较函数：
   - 删除 `isNewerClientVersion(String,String)`，换成 `isNewerClientVersion(Long incoming, Long current)`
   - `incoming == null`：抛 `ILLEGAL_PARAMETER`
5. 改 `ContentService.syncDraft()`：覆盖前先做版本比较

**验收**

- 传 `"clientVersion":"abc"`：反序列化直接失败（或返回 400/500），所以你必须同时改接口文档与客户端
- 传 `clientVersion=null`：返回 `0002`
- 传 `clientVersion` 小于服务端：返回 `STALE_VERSION`

### 1.6 「syncTime 被填成了 draftId」

**现状定位**

- `ContentController.syncDraft()`：
  - `syncTime(vo.getId())`，而 `vo.id` 是 draftId
  - `serverVersion(vo.getMessage())`，而 message 是 `"serverVersion-" + clientVersion`

**要改成什么**

- `syncTime` = 服务端落库时间（毫秒时间戳）
- `serverVersion` = 服务端接受并落库后的版本号（就是 draft.clientVersion）

**具体步骤（推荐：别再滥用 OperationResultVO）**

1. 新建一个专用返回 VO（domain 层）：
   - 路径：`project/nexus/nexus-domain/src/main/java/cn/nexus/domain/social/model/valobj/DraftSyncVO.java`
   - 字段：`Long serverVersion; Long syncTime;`
2. 修改 `IContentService.syncDraft(...)` 返回 `DraftSyncVO`
3. `ContentService.syncDraft(...)` 返回：
   - `serverVersion = entity.getClientVersion()`
   - `syncTime = entity.getUpdateTime()`（保存后取 now）
4. `ContentController.syncDraft()` 组装 DTO：
   - `serverVersion = String.valueOf(vo.getServerVersion())`
   - `syncTime = vo.getSyncTime()`

**验收**

- 返回的 `syncTime` 必须接近当前时间（允许有秒级误差）
- 返回的 `serverVersion` 必须等于你提交的 `clientVersion`

### 1.7 「新发帖没有稳定 postId → 锁/幂等失效 → 并发两条帖」

**现状定位**

- `ContentService.publish()`：
  - `Long targetPostId = postId == null ? socialIdPort.nextId() : postId;`
  - 所以 `postId=null` 每次都会新建 ID

**要改成什么（硬规则）**

- **禁止** `postId=null` 的 publish
- 新发帖只能走：先建草稿拿 `draftId`，再 publish 传 `postId=draftId`

**具体步骤**

1. 改 `ContentService.publish()`：
   - 如果 `postId == null`：直接抛 `ILLEGAL_PARAMETER`，信息写清楚：`postId 不能为空，请先创建草稿拿 postId`
2. 改 `ContentController.publish()`：
   - 不做额外逻辑，只透传；让服务端报错信息返回即可
3. 更新接口文档（必须写清楚调用顺序）：
   - `PUT /content/draft` → 得到 `draftId`
   - `POST /content/publish` → 传 `postId=draftId`

**验收**

- 并发 2 次 publish（同一个 postId）：
  - 必须只落 1 次成功版本（另一次复用 attempt 或返回进行中）
- 并发 2 次 publish（postId=null）：
  - 两次都必须失败（0002）

### 1.8 「风控异步回写只按状态推进 → 套错版本」

**现状定位**

- 发布 REVIEW 时：
  - `publish()` 把 `content_post.status=1(PENDING_REVIEW)`，并把 `version_num` +1
  - Attempt 记录了 `publishedVersionNum=newVersion`
- 人审回写在：`ContentService.applyRiskReviewResult(attemptId, ...)`
  - `contentRepository.updatePostStatus(postId, newStatus, expectedStatus=PENDING_REVIEW)`  
    **只看 status，不看 version，不看 attemptId**

**要改成什么**

- 回写必须同时满足：
  - post.status 仍是 `PENDING_REVIEW`
  - post.version_num == attempt.publishedVersionNum  
  （用版本号绑定“同一版内容”，不用额外字段就能解决）

**具体步骤**

1. 新增 DAO 更新方法（MyBatis）：
   - 在 `IContentPostDao` 增加 `updateStatusIfMatchAndVersion(postId, status, expectedStatus, expectedVersion)`
   - 在 `ContentPostMapper.xml` 增加：
     - `WHERE post_id=? AND status=? AND version_num=?`
2. 在 `IContentRepository` 增加方法：
   - `boolean updatePostStatusIfMatchVersion(Long postId, Integer status, Integer expectedStatus, Integer expectedVersion)`
3. 在 `ContentRepository` 实现它，成功后 `invalidatePostCache(postId)`
4. 修改 `ContentService.applyRiskReviewResult()`：
   - 取 `expectedVersion = attempt.getPublishedVersionNum()`
   - 用新方法推进状态
   - 如果推进失败：记录 warn，并返回 attempt 当前状态（不要硬报错）

**验收**

- 制造场景：同一个 post 连续 2 次触发 REVIEW（版本 1、2），先回写版本 1 的 PASS
  - 期望：版本 1 的 PASS **不能**把版本 2 发布
  - 最终只有“最新 attempt 对应版本”能推进到 PUBLISHED

### 1.9 「定时发布：必须绑定 postId；设置后禁止再编辑内容」

**现状定位**

- `ContentService.executeSchedule()` / `processSchedules()`：
  - 调用 `publish(null, ...)` → 每次生成新 postId（必然多发）
- `ContentService.updateSchedule()`：
  - 允许改 `contentData` → 等于“定时后还能改内容”（你已明确不允许）

**要改成什么（你已确认的规则）**

1. 定时任务必须绑定固定 `postId`（也就是 draftId）。
2. 一旦创建定时任务：
   - **禁止再编辑草稿或 post 内容**
   - 想编辑：必须先取消/删除该定时任务

**具体步骤（一步到位，不做半吊子兼容）**

1. 改表：`content_schedule` 增加 `post_id`（要能落地执行，不靠“脑补”）：

CREATE INDEX idx_content_schedule_post_status

-- 迁移：旧数据没有 postId，直接取消（否则你无法保证“不意外多发”）
UPDATE content_schedule
SET status = 3,
    is_canceled = 1,
    last_error = 'schema_migration: legacy schedule without post_id'
WHERE status = 0 AND post_id = 0;

<select id="selectActiveByPostId" resultMap="ContentScheduleMap">

   - `project/nexus/nexus-domain/src/main/java/cn/nexus/domain/social/adapter/repository/IContentRepository.java`

   - `project/nexus/nexus-domain/src/main/java/cn/nexus/domain/social/service/IContentService.java` 与 `project/nexus/nexus-domain/src/main/java/cn/nexus/domain/social/service/ContentService.java` 的 `schedule(...)`

     - `postId==null`：抛 `AppException(ResponseCode.ILLEGAL_PARAMETER, "请先创建草稿拿到 postId")`
     - 校验草稿归属：`draft.userId == userId`
     - 幂等 token 改为：`hash(userId + ":" + postId + ":" + publishTime)`
4. 定时后加“内容锁”（一处实现，多处复用，别散落 if/else）：
   - 在 `ContentService` 新增私有方法：`assertNotScheduled(postId)`

     - `active != null`：抛 `AppException(ResponseCode.CONFLICT, "已设置定时发布，如需编辑请先取消定时任务")`

     - `private OperationResultVO publishInternal(Long postId, Long userId, String text, String mediaInfo, String location, String visibility, List<String> postTypes)`
     - 这个私有方法里放“原本 publish 的全部逻辑”（不要改行为）

     - `ContentService.syncDraft(...)`（postId = draftId）
     - `ContentService.publish(...)`（HTTP 接口入口）：先 `assertNotScheduled(postId)`，再调用 `publishInternal(...)`

5. 改执行 schedule（`executeSchedule`/`processSchedules`）：
   - 必须调用：`publishInternal(task.getPostId(), ...)`，严禁再出现 `publish(null, ...)`
   - 发布内容来源（固定规则，不留口子）：
     1) 取草稿：`draft = contentRepository.findDraft(task.getPostId())`
     2) `draft == null`：把 schedule 标记 FAIL（并告警），不要“临时新建 post”
     3) 调用（参数按下面这行写死，不要留 `...`）：
        - `publishInternal(task.getPostId(), task.getUserId(), draft.getDraftContent(), draft.getMediaIds(), null, "PUBLIC", null)`
6. DTO 返回值补齐（让客户端能“知道自己锁的是哪条 post”）：
   - `ScheduleContentResponseDTO` 增加 `postId`

**验收**

1. 创建定时任务后：
   - 调 `PATCH /api/v1/content/draft/{draftId}`（同 draftId=postId）→ 返回 `0409`，提示“先取消定时”
   - 调 `POST /api/v1/content/publish`（同 postId）→ 返回 `0409`，提示“先取消定时”
   - 调 `PATCH /api/v1/content/schedule`（如果还带 contentData）→ 返回 `0409`，提示“先取消定时”
2. 取消定时任务后：以上 3 个操作恢复可用。
3. 同一个 taskId 被重复投递 2 次：只能生成同一个 postId 的一次发布（不会新增第二条 post）。

### 1.10 「delete 没跟 publish 用同一套并发规则 → 删了又活」

2. 修改 `project/nexus/nexus-infrastructure/src/main/java/cn/nexus/infrastructure/adapter/social/repository/ContentRepository.java`

     2) `private void invalidatePostCache(Long postId)`：现有只清 L1，这里改成 **L1 + Redis 都删**
     3) `private ContentPostEntity parsePostCache(String json)`：json->entity；遇到异常返回 null
3. 改 `ContentRepository.listPostsByIds(List<Long> postIds)` 的取数顺序（必须按这个顺序）：

   2) 对 missIds：批量走 Redis（一次 multiGet）
      - 命中 `"NULL"`：直接当不存在（不要回表）
      - 命中 json：反序列化成 `ContentPostEntity`，补进 resultById，并按 hotkey 规则回填 L1

      - DB 仍 miss：把 `"NULL"` 批量写回 Redis，TTL=30s（负缓存）
   4) 最后按入参顺序组装返回（保持现有行为不变）
4. 验收（你关心的是“减少 DB 命中”，所以验收要看这个）：
   - 压测 popular feed（同一批热门 postId 重复出现）：

### 1.29 「bitmap 替换每个 post/comment 的 set 来记录点赞用户」

**现状定位**

1. 真实实现：`project/nexus/nexus-infrastructure/src/main/java/cn/nexus/infrastructure/adapter/social/port/ReactionCachePort.java`
   - 目前 Lua 用的是 `SADD/SREM/SCARD`（Redis SET 存 userId 字符串）
   - 查询 state 用的是 `SISMEMBER`

**明确前提（不满足就先别改）**

- `userId` 必须是 **非负整数**（bitmap 下标不能是负数）
- `userId` 分布不能“极端稀疏”（否则分片 key 数量会爆）

**要改成什么（固定方案，不留选项）**

- 用 **分片 bitmap** 记录“某个 userId 是否点赞过某个 target”
- 计数仍用 `cntKey`（Lua 内只在状态变化时 `INCR/DECR`），避免线上跑 `BITCOUNT` 扫全量

**具体步骤（写到能直接改代码的粒度）**

1. Key 设计（必须照抄，保证 Redis Cluster 下安全）：
   - shard 大小：`BIT_SHARD_SIZE = 1_000_000`
   - bitmap key：`interact:reaction:bm:` + `target.hashTag()` + `:` + `shard`
     - 示例：`interact:reaction:bm:{POST:90001:LIKE}:0`
   - shard/offset：
     - `shard = userId / BIT_SHARD_SIZE`
     - `offset = userId % BIT_SHARD_SIZE`
   - 下面这些 key **保持不变**（仍使用同一个 `target.hashTag()`）：
     - `interact:reaction:cnt:` + `target.hashTag()`
     - `interact:reaction:ops:` + `target.hashTag()`
     - `interact:reaction:ops:processing:` + `target.hashTag()`
     - `interact:reaction:sync:` + `target.hashTag()`
2. 改 `ReactionCachePort.applyAtomic(...)`（只改这一处“写入口”，别到处补丁）：
   - 计算 `shard/offset`，得到 `bmKey`
   - Lua keys 改成 4 个（顺序固定）：
     1) `bmKey`
     2) `cntKey(target)`
     3) `opsKey(target)`
     4) `syncKey(target)`
   - Lua argv（顺序固定）：
     1) `userId`（字符串）
     2) `desiredState`（"1" 或 "0"）
     3) `offset`（数字字符串）
     4) `syncTtlSec`
3. 把 `ReactionCachePort` 里的 `LUA_APPLY_ATOMIC` 替换为下面这版（完整替换，不要混着改）：

```lua
local bmKey = KEYS[1]
local cntKey = KEYS[2]
local opsKey = KEYS[3]
local syncKey = KEYS[4]

local userId = ARGV[1]
local desiredState = ARGV[2]
local offset = tonumber(ARGV[3])
local syncTtlSec = tonumber(ARGV[4])

local delta = 0

-- cntKey 不存在时：先设成 0（避免 DECR 直接变负数）；后续读侧会从 DB count 表回填纠正
local current = redis.call('GET', cntKey)
if (not current) then
  redis.call('SET', cntKey, '0')
end

if desiredState == '1' then
  local old = redis.call('SETBIT', bmKey, offset, 1)
  if old == 0 then
    redis.call('INCR', cntKey)
    delta = 1
  end
else
  local old = redis.call('SETBIT', bmKey, offset, 0)
  if old == 1 then
    redis.call('DECR', cntKey)
    delta = -1
  end
end

-- last-write-wins：延迟落库时按 userId 覆盖即可
redis.call('HSET', opsKey, userId, desiredState)

-- 只在首次置 pending 时返回 1，避免重复投递延迟任务
local firstPending = redis.call('SET', syncKey, 'PENDING', 'NX', 'EX', syncTtlSec)

-- 防御：cntKey 可能被误删/误写，出现负数时拉回到 0（并让后续同步再对齐 DB）
local updated = tonumber(redis.call('GET', cntKey) or '0')
if updated < 0 then
  updated = 0
  redis.call('SET', cntKey, '0')
end

local fp = 0
if firstPending then fp = 1 end
return {updated, delta, fp}
```

4. 改 `ReactionCachePort.getState(...)`（查询是否点赞）：
   - 由 `SISMEMBER(setKey, userId)` 改成 `GETBIT(bmKey, offset) == 1`
   - Java 层用 `stringRedisTemplate.opsForValue().getBit(bmKey, offset)` 实现
5. 改 `ReactionCachePort.redisGetCntOrRebuild(...)`（cntKey 丢失时的确定策略：回表）：
   - 不再用 `SCARD`（因为我们不再有 set）
   - 做法（必须按这个）：
     1) `GET cntKey` 成功且 `>=0` → 直接返回
     2) 否则：**从 DB 的 `interaction_reaction_count` 回表读 count**，并 `SET cntKey` 回填后返回
   - 为了能回表读取，你需要补齐 DAO：
     - `project/nexus/nexus-infrastructure/src/main/java/cn/nexus/infrastructure/dao/social/IInteractionReactionCountDao.java` 增加方法：
       - `Long selectCount(String targetType, Long targetId, String reactionType);`
     - `project/nexus/nexus-infrastructure/src/main/resources/mapper/social/InteractionReactionCountMapper.xml` 增加对应 SQL（照抄）：

```xml
<select id="selectCount" resultType="java.lang.Long">
    SELECT count
    FROM interaction_reaction_count
    WHERE target_type = #{targetType}
      AND target_id = #{targetId}
      AND reaction_type = #{reactionType}
</select>
```
6. 验收（必须做三组用例）：
   1) 单用户：点赞 → 取消点赞 → 再点赞：state/计数都正确，delta 分别是 +1/-1/+1
   2) 并发：同一个 userId 并发点 10 次赞：最终计数只 +1（不会抖成 10）
   3) 人为删除 `cntKey`：再次点赞/查询时，会从 DB count 表回填，不会报错且计数不为负

### 1.30 「点赞延时落库：用 Redis ZSET + Lua 做指数退避重试（替换 RabbitMQ 固定 1s）」

**现状定位**

- 延迟触发：`project/nexus/nexus-domain/src/main/java/cn/nexus/domain/social/service/ReactionLikeService.java`
  - `reactionDelayPort.sendDelay(target, delayMs)`
- 现实现：RabbitMQ x-delay
  - 生产者：`project/nexus/nexus-trigger/src/main/java/cn/nexus/trigger/mq/producer/ReactionSyncProducer.java`
  - 消费者：`project/nexus/nexus-trigger/src/main/java/cn/nexus/trigger/mq/consumer/ReactionSyncConsumer.java`
  - 问题：失败后固定 `RETRY_DELAY_MS=1000`，会出现“1 秒 30 次”打桩式重试

**要改成什么（你已明确要用的实现方式）**

- 用 Redis 做一个“延迟队列”：
  - ZSET 的 score = “下次执行的时间戳(ms)”
  - Lua 脚本保证“取任务 + 标记处理中 + 失败重排”是原子操作
- 失败时按指数退避：`delay = min(base * 2^(attempt-1), max) + jitter`
- 同一个 target 同一时刻只允许一个 worker 处理（不再需要 `interact:reaction:lock:*`）

**具体步骤（完整落地，任何 agent 照做都能做出来）**

1. Redis key 设计（必须照抄；所有 key 共享同一个 hash-tag，保证 Redis Cluster 下 Lua 可运行）：
   - delay zset：`interact:reaction:sync:{rs}:delay`
   - processing zset：`interact:reaction:sync:{rs}:processing`
   - attempt hash：`interact:reaction:sync:{rs}:attempt`
   - dlq list：`interact:reaction:sync:{rs}:dlq`
   - member（任务 id）统一用：`{targetType}:{targetId}:{reactionType}`
     - 例：`POST:90001:LIKE`
2. 新增 4 个 Lua 脚本文件（路径固定）：
   - `project/nexus/nexus-trigger/src/main/resources/lua/reaction_sync/enqueue.lua`
   - `project/nexus/nexus-trigger/src/main/resources/lua/reaction_sync/poll.lua`
   - `project/nexus/nexus-trigger/src/main/resources/lua/reaction_sync/ack.lua`
   - `project/nexus/nexus-trigger/src/main/resources/lua/reaction_sync/fail.lua`

`enqueue.lua`（只保留**最早** dueTime；允许 processing 期间再次入队，用于“同步期间又产生新 ops”）：
```lua
local delayKey = KEYS[1]
local job = ARGV[1]
local dueMs = tonumber(ARGV[2])

local existing = redis.call('ZSCORE', delayKey, job)
if (not existing) or (tonumber(existing) > dueMs) then
  redis.call('ZADD', delayKey, dueMs, job)
end
return 1
```

`poll.lua`（把到期任务原子“搬”到 processing，并把 attempt+1；返回 job/attempt 成对数组）：
```lua
local delayKey = KEYS[1]
local processingKey = KEYS[2]
local attemptKey = KEYS[3]

local nowMs = tonumber(ARGV[1])
local limit = tonumber(ARGV[2])
local leaseMs = tonumber(ARGV[3])

-- 先把 processing 里“超时没 ack”的任务放回 delay（防止 worker 挂了任务永远卡住）
local expired = redis.call('ZRANGEBYSCORE', processingKey, '-inf', nowMs, 'LIMIT', 0, limit)
for i, job in ipairs(expired) do
  redis.call('ZREM', processingKey, job)
  local existing = redis.call('ZSCORE', delayKey, job)
  if (not existing) or (tonumber(existing) > nowMs) then
    redis.call('ZADD', delayKey, nowMs, job)
  end
end

local jobs = redis.call('ZRANGEBYSCORE', delayKey, '-inf', nowMs, 'LIMIT', 0, limit)
local out = {}
for i, job in ipairs(jobs) do
  -- 关键点：enqueue 允许“processing 期间再次入队”，所以 delay 里可能存在 processing 中的 job；必须跳过避免并发执行
  if (not redis.call('ZSCORE', processingKey, job)) then
    redis.call('ZREM', delayKey, job)
    redis.call('ZADD', processingKey, nowMs + leaseMs, job)
    local attempt = redis.call('HINCRBY', attemptKey, job, 1)
    table.insert(out, job)
    table.insert(out, tostring(attempt))
  end
end
return out
```

`ack.lua`（成功后清理 processing + attempt）：
```lua
local processingKey = KEYS[1]
local attemptKey = KEYS[2]
local job = ARGV[1]

redis.call('ZREM', processingKey, job)
redis.call('HDEL', attemptKey, job)
return 1
```

`fail.lua`（失败后：超过最大重试→进 dlq；否则重排到 delay）：
```lua
local processingKey = KEYS[1]
local delayKey = KEYS[2]
local attemptKey = KEYS[3]
local dlqKey = KEYS[4]

local job = ARGV[1]
local dueMs = tonumber(ARGV[2])
local maxAttempt = tonumber(ARGV[3])
local reason = ARGV[4]

local attempt = tonumber(redis.call('HGET', attemptKey, job) or '0')
redis.call('ZREM', processingKey, job)

if attempt >= maxAttempt then
  redis.call('HDEL', attemptKey, job)
  redis.call('LPUSH', dlqKey, job .. '|' .. tostring(attempt) .. '|' .. (reason or ''))
  return {0, attempt}
end

local existing = redis.call('ZSCORE', delayKey, job)
if (not existing) or (tonumber(existing) > dueMs) then
  redis.call('ZADD', delayKey, dueMs, job)
end
return {1, attempt}
```

3. 新增一个 Redis 版延迟端口实现（替换 RabbitMQ 生产者来入队）：
   - 新建类：`project/nexus/nexus-trigger/src/main/java/cn/nexus/trigger/redis/ReactionSyncDelayPortRedis.java`
   - 要求：`implements cn.nexus.domain.social.adapter.port.IReactionDelayPort`
   - `sendDelay(target, delayMs)`：
     1) job = `targetType:targetId:reactionType`
     2) dueMs = `System.currentTimeMillis() + max(0, delayMs)`
     3) 执行 `enqueue.lua`（keys：delayKey；argv：job、dueMs）
   - `sendToDLQ(rawMessage)`：直接 `LPUSH dlqKey rawMessage`（保底留痕）
4. 新增一个 worker（替换 RabbitMQ 消费者来“拉任务并执行”）：
   - 新建类：`project/nexus/nexus-trigger/src/main/java/cn/nexus/trigger/job/social/ReactionSyncZsetWorker.java`
   - 用 `@Scheduled(fixedDelay = 200)`（或做成配置项）循环执行：
     1) 执行 `poll.lua`（keys：delayKey、processingKey、attemptKey；argv：nowMs、limit=50、leaseMs=60000）
     2) 对返回的每个 (job, attempt)：
        - 解析 job → `ReactionTargetVO`
        - 调 `reactionLikeService.syncTarget(target)`
        - 成功：执行 `ack.lua`
        - 失败：计算 backoff，得到 `dueMs`，执行 `fail.lua`
     3) backoff 参数固定（写死或配置均可，但必须有默认值）：
        - `baseMs = 200`
        - `maxMs = 600_000`
        - `jitterMs = 200`（0~200 随机）
        - `maxAttempt = 30`
5. 禁止“双通道同时跑”（避免重复同步）：**固定用配置开关做互斥**（别删代码，先保证可回滚）
   - 统一配置键：`reaction.sync.mode`，取值只允许 `redis` 或 `rabbit`
   - 默认值：`redis`（也就是默认启用本节的 ZSET worker）
   - 改法（必须照做，确保 Spring 容器里只会出现 1 个 `IReactionDelayPort`）：
     1) 给 RabbitMQ 相关 Bean 全部加：
        - `@ConditionalOnProperty(name=\"reaction.sync.mode\", havingValue=\"rabbit\")`
        - 需要加的位置：
          - `project/nexus/nexus-trigger/src/main/java/cn/nexus/trigger/mq/config/ReactionSyncDelayConfig.java`

          - `project/nexus/nexus-trigger/src/main/java/cn/nexus/trigger/mq/consumer/ReactionSyncConsumer.java`
     2) 给 Redis 相关 Bean 全部加：

        - 需要加的位置：
