## 关键发现与决策点

### 当前实现分析

1. **`ReactionRepository.getCount()`** (第146-152行)：
   - 直接查询 DB 表 `interaction_reaction_count`
   - **不走缓存**
   - 这是 Feed 列表性能问题的根源

2. **`ReactionCachePort.getCount()`** (第147-166行)：
   - 有 L1 热点缓存 + single-flight
   - 走 `interact:reaction:cnt:*` Redis 缓存
   - 缓存 miss 时回表 `interaction_reaction_count` 重建

3. **`FeedCardStatRepository`**：
   - `getBatch()` 永远返回空 `Map.of()`
   - `getOrLoadBatch()` 有 single-flight，但因为 `getBatch()` 返回空，每次都走 loader
   - `saveBatch()` 是空实现（不做缓存）
   - 实际上是一个透传层

4. **`FeedCardAssembleService.rebuildStatCards()`** (第143-161行)：
   - 逐个 postId 调用 `reactionRepository.getCount(target)`
   - **每个 postId 都查一次 DB**（因为 `ReactionRepository.getCount()` 不走缓存）
   - 这导致"一页 20 条 Feed 就查 20 次 DB"的固定模式

### 核心问题

**问题根源**：`FeedCardAssembleService` 使用了错误的依赖。

- 当前：`rebuildStatCards()` → `reactionRepository.getCount()` → 直接查 DB
- 应该：`rebuildStatCards()` → `reactionCachePort.getCount()` → 走缓存

### 关键决策点 ⚠️

**决策 1**: `FeedCardAssembleService` 应该依赖哪个接口？

**选项 A**: 修改 `FeedCardAssembleService`，注入 `IReactionCachePort` 替代 `IReactionRepository`
- 优点：直接解决问题，走缓存
- 缺点：改变依赖关系，可能影响其他地方

**选项 B**: 修改 `ReactionRepository.getCount()`，让它内部调用 `ReactionCachePort.getCount()`
- 优点：不改变依赖关系
- 缺点：Repository 层调用 Port 层，违反分层原则

**选项 C**: 删除 `FeedCardStatRepository`，让 `FeedCardAssembleService` 直接调用 `reactionCachePort.getCount()`
- 优点：消除中间层，简化逻辑
- 缺点：需要修改 `FeedCardAssembleService` 的依赖

**我的建议**: 选项 C - 删除 `FeedCardStatRepository`，直接使用 `IReactionCachePort`

**理由**：
1. `FeedCardStatRepository` 当前是一个无用的透传层
2. 文档明确说"若保留 `FeedCardStatRepository`，它只能转调现有指标缓存"
3. 直接使用 `IReactionCachePort` 更清晰，避免多余抽象
