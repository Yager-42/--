# xiaohashu 项目简历逐条深挖稿

## 先给面试官的 1 分钟总述

这个项目是一个偏“小红书后端中台”的微服务练手项目，按业务域拆成了网关、认证、用户、笔记、用户关系、计数、评论、搜索、对象存储、KV 短文本、分布式 ID、数据对齐等服务。核心目标不是做一个纯 CRUD 系统，而是围绕高并发读写、缓存分层、异步解耦、最终一致性、搜索索引同步这些典型场景，练完整的服务拆分和中间件落地。

如果你面试时先总述，建议按这条主线讲：

4. 最后说计数与搜索为什么需要独立服务，以及为什么要做数据对齐。

这份稿子不是“漂亮话版本”，而是按源码可落地程度来写的。每个点都分成：

## 1. 网关服务

### 简历上怎么说

负责 Gateway 网关服务的搭建与开发，实现统一路由管理、负载均衡；使用 SaToken 实现网关统一鉴权；全局过滤器解析 Token，并将用户 ID 透传给下游服务。

### 业务上下文

微服务拆分之后，前端如果直接访问各个服务，会遇到几个问题：

1. 服务地址太多，前端要维护很多域名和端口。
2. 每个服务自己鉴权会重复开发。
3. 登录态和角色权限的校验逻辑会分散。
4. 下游服务经常需要知道“当前用户是谁”，但又不应该每个服务都去重复解析 Token。

所以网关的作用是做统一入口，把“路由转发、登录校验、角色权限校验、用户上下文透传”收口。

### 实际请求链路

以“用户调用一个需要登录的业务接口”为例：

1. 请求先进入 `xiaohashu-gateway`。
2. `SaTokenConfigure#getSaReactorFilter()` 会拦截所有路由。
3. 对 `/auth/login`、`/auth/verification/code/send` 放行，其余路由执行 `StpUtil.checkLogin()`。
4. 针对部分接口再做角色校验，比如 `/auth/logout` 需要 `common_user` 角色。
5. 理论上，鉴权通过后，`AddUserId2HeaderFilter#filter()` 会从 Sa-Token 上下文取出登录用户 ID。
6. 再把用户 ID 放入请求头，供下游服务读取。
7. 下游服务通过 `HeaderUserId2ContextFilter` 把请求头中的用户 ID 放入 `LoginUserContextHolder`。
8. 如果下游继续调用别的服务，`FeignRequestInterceptor#apply()` 再把用户 ID 放回 Feign 请求头，实现链路透传。

### 设计动机

1. 鉴权前置到网关，可以减少业务服务里的重复代码。
2. 用户上下文通过 Header 透传，比每个服务都解析 Token 更简单，也更统一。
3. 角色和权限数据放 Redis，由 `StpInterfaceImpl` 读取，避免每次查库。

### 技术原理补充

#### 原理说明

1. Gateway 本质上是反向代理加过滤器链，请求先经过统一入口，再按路由规则转发到具体服务。
2. 这里用的是 Sa-Token 的 Reactor 过滤器，核心是把登录校验和权限校验前置到响应式入口层。
3. 用户上下文透传走的是“Header -> Filter -> ThreadLocal -> Feign Interceptor”链路，这是一种典型的跨服务用户上下文传播方式。
4. Redis 存角色和权限，本质是在用缓存提升高频鉴权吞吐，避免每次权限判断都查库。

#### 原理类可能提问

1. 什么是反向代理，为什么网关适合做统一鉴权？
   答法：反向代理是客户端只访问一个统一入口，由代理服务器把请求转发给后端真实服务。网关正好位于所有请求最前面，所以最适合统一做鉴权、限流、路由和日志收口。
2. Reactor 模型和传统 Servlet 模型的过滤器有什么差别？
   答法：Servlet 更偏“一请求一线程”的同步阻塞模型，Reactor 更偏事件驱动和异步非阻塞模型。这里用 Reactor 过滤器，本质是把鉴权逻辑放在响应式链路前端执行。
3. ThreadLocal 为什么适合做请求级上下文？为什么要在 finally 里清理？
   答法：因为一次请求在当前线程内执行时，ThreadLocal 可以很方便地存当前用户 ID 这类上下文。但线程池会复用线程，如果不清理，就可能把上一个请求的数据污染到下一个请求。
4. Feign 拦截器为什么能实现跨服务上下文透传？
   答法：因为 Feign 在真正发 HTTP 请求前会经过 `RequestInterceptor`，所以可以把当前线程里的 userId 统一塞进请求头，让下游继续感知当前用户。

### 高频追问

#### Q1：网关为什么适合做统一鉴权？

答法：
因为它是所有请求的入口。统一鉴权的价值在于把“谁能访问什么接口”这件事在入口处就拦住，避免非法请求进入业务服务。另外，网关对下游服务是透明的，下游只关心用户 ID，不关心 Token 解析细节。

#### Q2：角色和权限数据怎么查？

答法：
`StpInterfaceImpl#getRoleList()` 和 `getPermissionList()` 会根据登录用户 ID 从 Redis 读用户角色，再批量读角色权限。这样做比每次查数据库更快，也更适合高频鉴权。

#### Q3：下游服务为什么不用自己解析 Token？

答法：
因为职责不应该分散。网关负责认证，下游负责业务。如果每个服务自己解析 Token，第一是重复开发，第二是修改认证逻辑时改动面太大，第三是更难统一治理。

### 谨慎表述点

1. 源码里 `AddUserId2HeaderFilter#filter()` 虽然构造了 `newExchange`，但最后返回的是 `chain.filter(exchange)`，不是 `chain.filter(newExchange)`。也就是说，“透传 userId 到下游”这件事从代码看有实现意图，但当前实现有瑕疵。
2. 如果面试官追到源码层，建议你说：
   我设计上是想在网关透传用户 ID，配套也写了下游的 Header 解析和 Feign 透传，但当前这个 filter 有一个小 bug，应该传递修改后的 `newExchange`。
3. 不要说“JWT 鉴权”。配置里 `sa-token.token-style` 是 `random-128`，不是 JWT。

## 2. 认证服务

### 简历上怎么说

负责认证服务的搭建与开发，基于 RBAC 模型，使用 SaToken 实现用户登录认证、登出、用户自注册、密码修改；支持账号登录、手机验证码登录；对接阿里云 SMS 服务，自定义线程池异步发送验证码，并实现短信接口防刷。

### 业务上下文

认证服务的核心目标是把“身份确认”和“登录态发放”单独拆出来，避免用户服务又管资料又管登录。这样用户资料服务更像一个账号数据服务，认证服务只负责：

1. 校验登录凭证。
2. 发放登录态。
3. 调用用户服务做注册或更新密码。
4. 发送验证码。

### 实际请求链路

#### 验证码登录链路

1. 前端先调用验证码发送接口。
2. `VerificationCodeServiceImpl#send()` 先检查 Redis 中是否已有未过期验证码。
3. 如果有，直接拒绝，防止用户短时间重复发送。
4. 如果没有，生成 6 位验证码。
5. 把发送短信动作丢到线程池异步执行，调用 `AliyunSmsHelper`。
6. 同时把验证码写入 Redis，过期时间 3 分钟。
7. 用户提交手机号和验证码后，`AuthServiceImpl#loginAndRegister()` 读取 Redis 中的验证码做校验。
8. 验证通过后，调用用户服务 `registerUser(phone)`，如果用户已存在就直接返回原用户 ID，不存在则注册。
9. 最后使用 `StpUtil.login(userId)` 发放登录态并返回 token。

#### 密码登录链路

1. `AuthServiceImpl#loginAndRegister()` 先通过用户服务根据手机号查用户。
2. 取回数据库里已加密的密码。
3. 使用 `PasswordEncoder.matches()` 校验明文密码和加密串是否匹配。
4. 通过后执行 `StpUtil.login(userId)`。

#### 修改密码链路

1. 认证服务接收新密码。
2. `PasswordEncoderConfig` 中使用 `BCryptPasswordEncoder` 做加密。
3. 把加密后的密码传给用户服务更新。

### 设计动机

1. 登录和账号信息修改分服务，职责更清晰。
2. 验证码写 Redis 天然适合做短期有效和简单防刷。
3. 短信发送异步化，避免接口阻塞在第三方服务上。
4. 密码统一在认证侧加密，用户侧只存密文。

### 技术原理补充

#### 原理说明

1. 当前认证模式不是 JWT 自描述 Token，而是服务端维护登录态、客户端持有随机 Token 的模式。
2. Redis TTL 非常适合验证码这种短生命周期数据，过期控制不需要业务代码额外扫表。
3. BCrypt 的核心是随机盐加慢哈希，同样的明文多次加密结果不同，但校验仍然可逆向验证。
4. 线程池异步发送短信，本质上是把第三方 IO 调用从主业务线程剥离出去，降低接口响应时间。

#### 原理类可能提问

1. JWT 和服务端 Session/Token 模式有什么本质区别？
   答法：JWT 是自包含 Token，很多信息放在令牌里；服务端 Session/Token 模式是服务端保存登录态，客户端只拿一个随机令牌。前者更去中心化，后者更容易做失效控制和服务端统一治理。
2. BCrypt 为什么比 MD5、SHA 系列更适合存密码？
   答法：因为 BCrypt 是慢哈希并且自带随机盐，同样密码每次加密结果都不同，暴力破解成本更高。MD5、SHA 更适合做摘要，不适合直接存用户密码。
3. Redis 为什么适合存验证码，不适合直接存长期账号主数据？
   答法：验证码天然是短生命周期、高并发、过期即失效的数据，非常适合 Redis TTL。账号主数据则更需要事务、一致性和持久化治理，所以更适合关系型数据库。
4. 异步发短信为什么能提速，但为什么还要关注发送失败重试？
   答法：异步能把第三方短信 IO 从主线程移走，降低接口 RT；但异步不代表可靠性自动解决，如果短信通道偶发失败，仍然需要重试或失败兜底机制。

### 高频追问

#### Q1：为什么验证码登录路径里直接带注册？

答法：
这是一个典型的“登录即注册”设计，适合手机号体系。用户如果第一次登录，不需要先手动注册，再登录，直接在认证成功后调用用户服务完成注册，降低用户操作成本。

#### Q2：密码为什么用 BCrypt？

答法：
因为 BCrypt 自带随机盐，能有效防止彩虹表攻击，而且 Spring Security 对它支持很成熟。项目里通过 `BCryptPasswordEncoder` 做加密和校验。

#### Q3：短信防刷是怎么做的？

答法：
当前实现是手机号维度的防刷。发送验证码前先检查 Redis 是否已有未过期验证码，有就直接拒绝。它能防“同一手机号短时间频繁发送”，但如果要再往上做，还可以加 IP 维度、设备维度、图形验证码、滑块验证和发送次数上限。

### 谨慎表述点

1. 不要说“JWT 登录认证”。仓库配置是 `random-128` token，不是 JWT。
2. “支持账号登录”要谨慎。源码里密码登录是“手机号 + 密码”，不是用户名账号体系。
3. “短信接口防刷”目前只是基于手机号 + Redis TTL 的轻量限流，不是复杂风控。

## 3. 分布式 ID 生成服务

### 简历上怎么说

负责整合美团 Leaf 源码，搭建分布式 ID 生成服务，以 Feign 接口形式对外提供号段模式 ID 和雪花算法 ID。

### 业务上下文

微服务环境下不能再依赖数据库自增主键，因为：

1. 多服务写不同库时难统一。
2. 跨库合并数据时可能冲突。
3. 高并发场景下中心化自增容易成为瓶颈。

所以这里单独做了一个 ID 服务，对外提供统一的业务 ID。

### 实际请求链路

1. 业务服务通过 Feign 调用分布式 ID 服务。
2. 请求进入 `LeafController`。
3. 如果走 `/id/segment/get/{key}`，就使用 `SegmentService` 和 `SegmentIDGenImpl`。
4. 如果走 `/id/snowflake/get/{key}`，就使用 `SnowflakeService` 和 `SnowflakeIDGenImpl`。

### 设计动机

#### 号段模式

1. `SegmentIDGenImpl` 会从数据库申请一段号段到本地缓存。
2. 本地发号时直接在内存里原子递增。
3. 当前号段快用完时，后台线程预加载下一段。
4. 这样把大部分请求都变成本地内存操作，性能高。

#### 雪花模式

1. `SnowflakeIDGenImpl` 通过 ZK 协调 workerId。
2. 生成 ID 时由时间戳、workerId、序列号拼接。
3. 适合对吞吐要求高、且希望一定程度趋势递增的场景。

### 技术原理补充

#### 原理说明

1. 号段模式的核心是“批量申请、内存发号”，把数据库从每次发号参与者变成低频号段分配者。
2. `SegmentBuffer` 双 buffer 的本质是当前号段对外发号，下一号段后台预热，减少切段时抖动。
3. 雪花算法由时间戳位、机器位、序列号位组成，所以具备趋势递增和多机唯一的特点。
4. ZooKeeper 在这里承担的是 workerId 协调角色，防止不同节点生成相同机器号。

#### 原理类可能提问

1. 双 buffer 为什么能降低号段切换时延？
   答法：因为当前 buffer 在发号时，下一 buffer 可以后台提前装载好，下一个号段切换时不需要临时去 DB 申请，所以切换更平滑。
2. 雪花算法为什么会受时钟回拨影响？
   答法：因为雪花 ID 的高位里包含时间戳，如果机器时间倒退，就可能生成比之前更小甚至冲突的 ID，所以必须检测和处理回拨。
3. 号段模式和数据库自增主键的差别是什么？
   答法：数据库自增是每次发号都依赖数据库；号段模式是一次取一段号，后续在内存里连续发，大部分请求不再打库，吞吐高很多。
4. 为什么 ZooKeeper 适合做分布式 workerId 分配？
   答法：因为它擅长做分布式协调和节点唯一标识分配，能帮助多节点拿到不冲突的 workerId，这正是雪花算法最需要解决的问题之一。

### 高频追问

#### Q1：号段模式和雪花模式怎么选？

答法：
号段模式的优点是稳定、发号逻辑简单、对时钟不敏感；缺点是依赖数据库做号段分配。雪花模式吞吐更高，趋势递增更明显，但依赖机器时钟和 workerId 分配。业务上我会根据场景选择，比如用户 ID、小哈书号适合号段或稳定分配类场景，笔记/评论这种高频写可以用雪花。

#### Q2：号段模式为什么快？

答法：
因为真正发号时不打数据库，只是在本地 `SegmentBuffer` 里做内存自增。数据库只在“申请新号段”时访问一次。

#### Q3：雪花算法怎么处理时钟回拨？

答法：
源码里 `SnowflakeIDGenImpl#get()` 对小范围回拨会短暂等待，对较大回拨直接返回异常状态，避免生成重复 ID。

### 谨慎表述点

1. “整合美团 Leaf 源码”这个说法成立，目录结构和实现明显来源于 Leaf。
2. “JMeter 压测 22000+/s、平均 4ms、日 19 亿+ 次”这组数字仓库里没有压测脚本或报告。面试时只能说“我做过压测，量级大致在这个范围”，不要说得像源码内有证据。

## 4. OSS 对象存储服务

### 简历上怎么说

负责对象存储服务搭建与开发，存储图片、视频等非结构化数据；使用工厂模式 + 策略模式实现存储扩展，已支持阿里云 OSS、Minio；通过配置切换存储实现。

### 业务上下文

用户头像、背景图、笔记图片、视频都不适合直接存数据库。对象存储服务的核心目标是统一文件上传出口，屏蔽底层存储实现差异。

### 实际请求链路

1. 业务服务调用 OSS 服务上传文件。
2. `FileServiceImpl#uploadFile()` 接收文件。
3. 内部依赖 `FileStrategy`。
4. `FileStrategyFactory#getFileStrategy()` 根据 `storage.type` 选择 `MinioFileStrategy` 或 `AliyunOSSFileStrategy`。
5. 实际上传完成后返回访问 URL。

### 设计动机

1. 用策略模式隔离不同存储厂商的 SDK 细节。
2. 业务侧只关心“传文件，拿 URL”，不关心底层到底是阿里云还是 Minio。
3. 通过配置切换策略，方便开发环境和生产环境使用不同对象存储。

### 技术原理补充

#### 原理说明

1. 策略模式的核心是定义统一行为接口，把不同厂商的差异实现封装到不同策略类中。
2. 工厂模式负责屏蔽策略选择逻辑，业务层不需要知道当前到底实例化的是哪种实现。
3. `@RefreshScope` 的本质是让配置变化后，相关 Bean 能重新初始化，从而切换当前策略。
4. Multipart 上传本质是 HTTP 分段表单传输，服务端接收到文件流后再转存对象存储。

#### 原理类可能提问

1. 工厂模式和策略模式为什么经常一起出现？
   答法：策略模式解决“行为如何抽象”，工厂模式解决“具体用哪种策略”。一个负责定义统一能力，一个负责选择实现，所以很常一起出现。
2. 为什么对象存储适合放图片和视频，而不是数据库 BLOB？
   答法：图片和视频体积大、下载频繁，放数据库会让主库膨胀且影响备份与查询。对象存储更适合大文件、静态资源和海量非结构化数据。
3. `@RefreshScope` 解决的是配置热更新的什么问题？
   答法：它解决的是配置变了但 Bean 还是老值的问题。加了它以后，配置刷新时相关 Bean 可以重新初始化，新的配置才能真正生效。
4. MinIO 和阿里云 OSS 在抽象层面为什么能统一成一个接口？
   答法：因为它们对业务来说都提供了相似能力：上传对象、生成对象路径、按 bucket 管理文件。业务层关心的是“上传文件”，不关心底层 SDK 差异。

### 高频追问

#### Q1：为什么要用策略模式？

答法：
因为阿里云 OSS 和 Minio 的初始化方式、上传 API、返回 URL 逻辑不一样。如果直接把判断写在业务里，后面接更多厂商会越来越乱。策略模式可以把“上传行为”抽象成统一接口，把差异性封装到实现类里。

#### Q2：工厂模式在这里起什么作用？

答法：
工厂负责根据配置决定当前激活哪一个策略实现，相当于把“实例选择逻辑”从业务代码中再抽走一层。

### 谨慎表述点

1. 源码里确实用了“工厂 + 策略”的组合。
2. 但“动态注册策略实现类 Bean 到 Spring 容器中”这个说法要收一点。当前代码更准确的描述是：
   通过 `@RefreshScope` 和配置项 `storage.type`，在工厂中返回不同的 `FileStrategy` Bean。
3. 它不是一个通用的“运行时自动扫描并注册任意策略实现类”的框架级方案。

## 5. KV 短文本键值存储服务

### 简历上怎么说

负责 KV 键值存储服务搭建与开发，用于存储笔记正文、评论内容等短文本内容。

### 业务上下文

笔记元数据和正文、评论元数据和正文，访问模式不完全一致。正文通常体积更大、读写频率和查询维度也不一样，把它们拆出来有几个好处：

1. 元数据表更轻。
2. 列表查询不需要总是扫大字段。
3. 正文可用更适合键值/宽表的存储承接。

### 实际请求链路

#### 笔记正文

1. 笔记服务发布笔记时生成 `contentUuid`。
2. 调用 `KeyValueRpcService.saveNoteContent()`。
3. KV 服务进入 `NoteContentServiceImpl#addNoteContent()`。
4. 使用 `NoteContentRepository` 把正文保存到 Cassandra。

#### 评论正文

1. 评论服务批量消费评论发布消息。
2. 在 `Comment2DBConsumer` 里把有正文的评论提取出来。
3. 调用 `keyValueRpcService.batchSaveCommentContent(commentBOS)`。
4. KV 服务进入 `CommentContentServiceImpl#batchAddCommentContent()`。
5. 通过 `CassandraTemplate.batchOps().insert()` 批量写入。

### 设计动机

1. 把正文从主业务表拆走，减少主表宽度。
2. 评论正文按 `(noteId, yearMonth, contentId)` 设计主键，有利于按笔记维度做批量读取。
3. 评论正文场景明确支持批量写、批量查，更贴近评论分页读取的访问模式。

### 技术原理补充

#### 原理说明

1. Cassandra 是宽列模型，擅长按主键快速读写和横向扩展，不擅长复杂 Join。
2. 复合主键本质是在做分区设计，目标是让热点更均匀、查询路径更稳定。
3. 批量接口能减少 RPC 次数、序列化成本和存储往返，适合评论列表这种天然批量场景。
4. 正文和元数据分离，本质是把大字段从热点结构化表中拆出去。

#### 原理类可能提问

1. 宽列存储和关系型数据库在设计思路上有什么不同？
   答法：关系型数据库强调规范化、事务和灵活查询；宽列存储更强调按访问模式设计主键，追求高吞吐和横向扩展。它通常是“先定查询，再定表”。
2. 分区键为什么会直接影响性能和热点分布？
   答法：因为分区键决定一条数据落到哪个分区。如果分区设计不好，热点数据会集中在少数分区，导致单点过热和吞吐不均。
3. 为什么批量查询通常比逐条查询更高效？
   答法：因为它能减少 RPC 次数、网络往返、序列化开销和存储端查询次数，把很多固定成本合并掉。
4. Cassandra 为什么不适合临时做复杂多表关联？
   答法：因为它不是为 Join 场景设计的，更多是按主键直达读写。复杂关联查询通常要靠业务层提前冗余或单独的检索系统来解决。

### 高频追问

#### Q1：为什么笔记正文和评论正文都单独拆出去？

答法：
因为正文字段通常是大字段，不适合一直跟元数据耦合在一张热点业务表里。拆出去以后，主流程查列表、查摘要时更轻，需要正文时再按 key 定位。

#### Q2：评论正文为什么做批量接口？

答法：
因为评论写入和评论列表读取都天然容易成批出现。比如 MQ 批量消费评论消息、评论分页一次取多条，这时候批量写 KV、批量查 KV 都能减少网络开销和存储端压力。

### 谨慎表述点

1. 代码中实际落地的是 Spring Data Cassandra。
2. 仓库里没有看到你自己集成 RocksDB 的实现证据，所以不要说“基于 RocksDB 存储引擎上构建的 Apache Cassandra”是你亲自改造出来的。
3. 面试建议改成：
   使用 Cassandra 承载笔记正文和评论正文等短文本内容，并为评论场景设计了批量读写接口。

## 6. 用户服务

### 简历上怎么说

负责用户服务设计与实现，提供用户信息修改、注册、查询、密码更新等能力；使用 Redis + Caffeine 构建二级缓存，支持高并发读，并通过随机过期、空值缓存等方式防止缓存雪崩、穿透、击穿。

### 业务上下文

用户服务是很多业务的底座。笔记详情、评论列表、粉丝列表都会频繁查用户头像、昵称、简介，所以这个服务的读多写少特征很明显，适合做二级缓存。

### 实际请求链路

#### 注册

1. 认证服务调用用户服务注册。
2. `UserServiceImpl#register()` 先按手机号查是否已存在。
3. 不存在则调用分布式 ID 服务生成用户 ID 和小哈书号。
4. 写入 `t_user`。
5. 再写 `t_user_role` 分配默认角色。
6. 最后把角色列表写入 Redis，供网关鉴权读取。

#### 查询用户信息

1. `findById()` 先查 Caffeine 本地缓存。
2. 本地没命中再查 Redis。
3. Redis 没命中再查数据库。
4. 查到后异步回填 Redis。
5. Redis 命中后异步回填本地缓存。
6. 若数据库也不存在，则把 `"null"` 写入 Redis 短期缓存，防缓存穿透。

#### 批量查询用户信息

1. 先构造 Redis key 列表。
2. 使用 `multiGet` 批量查 Redis。
3. 对缺失数据再批量查 DB。
4. 用 pipeline 批量回填 Redis。

### 设计动机

1. 本地 Caffeine 承担最热点数据，减少网络跳转。
2. Redis 作为分布式共享缓存，承担大部分跨实例读取。
3. `null` 缓存防止穿透。
4. 随机 TTL 打散过期时间，降低雪崩风险。
5. 批量查询用 `multiGet + pipeline`，减少 Redis 往返。

### 技术原理补充

#### 原理说明

1. Caffeine 是进程内缓存，访问速度快，但只在当前实例生效。
2. Redis 是分布式缓存，适合多实例共享，但会多一次网络 IO。
3. 空值缓存是典型防穿透手段，核心思想是“对不存在的数据也缓存一个短期结果”。
4. 随机 TTL 的本质是打散大批量 key 的失效时间，降低同一时间集中过期造成的雪崩。
5. `multiGet` 和 pipeline 都是在减少网络往返，只是一个偏批量读取，一个偏批量写回。

#### 原理类可能提问

1. 本地缓存和分布式缓存各自适合什么场景？
   答法：本地缓存适合极热点、极低延迟场景；分布式缓存适合多实例共享和统一失效控制。两者结合能同时兼顾速度和共享性。
2. 什么是缓存穿透、击穿、雪崩，它们的区别是什么？
   答法：穿透是查不存在的数据，击穿是热点 key 失效瞬间被打爆，雪崩是大量 key 同时失效。区别在于“打数据库的原因”不同。
3. 为什么空值缓存不能把 TTL 设得太长？
   答法：因为当前不存在不代表一直不存在，如果 TTL 太长，真实新数据写入后，缓存里的空值会让用户长时间看不到最新结果。
4. W-TinyLFU 这类本地缓存淘汰策略大概解决了什么问题？
   答法：它解决的是有限缓存空间下“哪些数据更值得留下来”的问题，通过兼顾访问频率和近期热度，提高热点命中率。

### 高频追问

#### Q1：为什么本地缓存和 Redis 要一起用？

答法：
本地缓存命中最快，适合极热点数据；Redis 适合跨实例共享。只用本地缓存会导致每个实例数据孤岛，只用 Redis 又比本地多一次网络 IO，所以结合起来比较平衡。

#### Q2：你怎么防缓存穿透？

答法：
数据库查不到的时候，不是直接返回，而是把 `"null"` 这种空值短期写入 Redis。这样后面的同类请求不会一直打数据库。

#### Q3：你怎么防缓存雪崩？

答法：
不是给所有 key 统一过期时间，而是“基础 TTL + 随机秒数”，把大量缓存的失效时间打散。

### 谨慎表述点

1. “防缓存击穿”更多是通过二级缓存和异步回填缓解，源码里没有互斥锁或 singleflight 这种强击穿治理。
2. `updateUserInfo()` 更新用户信息后没有看到同步清理本地缓存和 Redis 用户缓存的逻辑，这一点如果面试官追问一致性，要主动承认还有优化空间。
3. 密码加密实际发生在认证服务，用户服务负责持久化加密后的密码。

## 7. 笔记服务

### 简历上怎么说

负责笔记服务设计与实现，支持发布、编辑、查询、置顶、可见性修改、点赞、收藏等；通过 `CompletableFuture` 并发调用下游服务降低响应耗时；通过二级缓存支持详情高并发读；更新删除时用 RocketMQ 广播删除本地缓存；点赞接口通过布隆过滤器 + ZSET + MQ 异步落库支持高并发写。

### 业务上下文

笔记服务是内容核心。它既有“查详情”的高并发读，也有“点赞、收藏”的高并发写，还有“正文不在同库”的跨服务写一致性问题，所以是整个项目里设计最复杂的服务之一。

### 实际请求链路

#### 发布笔记

1. `publishNote()` 先校验笔记类型。
2. 图文笔记要求有图片且不超过 8 张，视频笔记要求有视频地址。
3. 先通过分布式 ID 服务生成笔记 ID。
4. 如果正文不为空，生成 `contentUuid`，先调用 KV 服务保存正文。
5. 再写笔记元数据到 `t_note`。
6. 如果元数据写入失败，再反向调用 KV 删除正文，做补偿。
7. 最后异步发送 `TOPIC_NOTE_OPERATE` 消息给后续计数/对齐等链路消费。

#### 查询笔记详情

1. 先查 Caffeine 本地缓存。
2. 再查 Redis。
3. 再查数据库。
4. 命中 DB 后，并发发两个 RPC：
   - 调用户服务取作者信息
   - 调 KV 服务取正文
5. 通过 `CompletableFuture.allOf()` 合并结果。
6. 组装成详情 VO 后异步写 Redis，再由 Redis 命中时异步回填本地缓存。

#### 更新/删除笔记

1. 先校验操作者是不是笔记作者。
2. 更新元数据。
3. 删除 Redis 详情缓存。
4. 发延迟消息做“延迟双删”。
5. 发广播消息删所有实例本地缓存。
6. 再更新或删除 KV 正文。

#### 笔记点赞

1. 先校验笔记存在。
2. 用布隆过滤器判断“用户是否可能点过赞”。
3. 如果布隆过滤器不存在，就查 DB 并初始化布隆过滤器。
4. 如果布隆过滤器判断已点赞，还会查 ZSET 或 DB 做二次确认，防止误判。
5. 通过 Lua 更新用户点赞 ZSET。
6. 发顺序 MQ 到消费端异步落库。
7. 消费端 `LikeUnlikeNoteConsumer` 会按用户分组、按 noteId 合并内存操作，只保留最终状态，再批量 upsert 到 DB。

### 设计动机

1. 详情查询拆成“元数据 + 用户信息 + 正文”并发拼装，减少串行等待。
2. 二级缓存适合读多写少的详情接口。
3. 笔记更新和删除时使用“删 Redis + 延迟双删 + 广播删本地缓存”来降低脏读概率。
4. 点赞/收藏不走同步落库，而是“Redis 快路径 + MQ 异步落库”，把写压力从 DB 挪开。
5. 布隆过滤器用于快速判断“基本不可能存在”，DB 只作为兜底确认。

### 技术原理补充

#### 原理说明

1. `CompletableFuture` 的核心价值是把相互独立的 RPC 并发执行，用等待最慢请求代替串行等待总和。
2. Bloom Filter 是概率型数据结构，结论是“不存在一定不存在，存在可能存在”。
3. Redis ZSET 适合存“带时间顺序的关系集合”，比如点赞时间、收藏时间。
4. 延迟双删是缓存一致性里常见的折中方案，用来尽量降低并发更新时脏缓存残留。
5. 本地缓存广播失效，本质是在解决多节点本地缓存天然不共享的问题。

#### 原理类可能提问

1. `CompletableFuture.allOf()` 为什么适合聚合多个下游调用？
   答法：因为它可以等待多个独立异步任务一起完成，特别适合“作者信息”和“正文内容”这种互不依赖的 RPC 聚合场景。
2. Bloom Filter 为什么会误判，但不会漏判？
   答法：因为真实存在的元素插入时一定把对应 bit 位置都置成 1，所以不会漏判；但不同元素可能把同一组 bit 也置成 1，所以会误判。
3. 为什么 ZSET 适合做最近点赞列表，而不是普通 Set？
   答法：Set 只能判断存在，不支持按时间排序；ZSET 可以用 score 存时间戳，所以既能判存在，又能按最近操作排序和分页。
4. 延迟双删到底在防什么问题？
   答法：它主要防的是并发更新时“删缓存和回源查询交错”导致旧值重新回到缓存里的问题，通过延迟再删一次来降低脏缓存残留概率。

### 高频追问

#### Q1：为什么发布笔记时先写 KV，再写元数据？

答法：
因为正文和元数据不在同一个存储里，做不了本地事务。我这里先写正文，再写元数据。如果第二步失败，就反向删正文做补偿，尽量保证两边一致。

#### Q2：`CompletableFuture` 带来的收益是什么？

答法：
笔记详情依赖用户服务和 KV 服务两个下游。如果串行调用，总耗时接近两个 RPC 相加；并行调用时，总耗时更接近耗时最长的那个 RPC，接口延迟会明显下降。

#### Q3：布隆过滤器为什么不能单独作为最终判断？

答法：
因为布隆过滤器有误判率，只能保证“不存在判断一定准，存在判断可能误判”。所以代码里在“判已点赞”时还会查 ZSET 或 DB 做二次确认。

### 谨慎表述点

1. 仓库里只看到布隆过滤器相关 Lua 和逻辑，没有 Roaring Bitmap 落地代码，所以不要说“后优化为 Roaring Bitmap”是当前项目源码能力。
2. 如果面试官问“为什么不用 Roaring Bitmap”，你可以说：
   这个项目当前是布隆过滤器方案，Roaring Bitmap 是我后续思考过的优化方向，不是当前仓库里已经落地的版本。
3. “高并发写”这个说法成立，但要落在“异步化、顺序消息、批量落库、内存合并”这些点上，不要只喊口号。

## 8. 用户关系服务

### 简历上怎么说

负责用户关系服务设计与实现；关注/取关通过 Redis + 顺序 MQ 异步落库支持高并发写；使用 Lua 脚本减少频繁 Redis 操作并保证原子性；粉丝列表、关注列表通过 Redis ZSET 缓存热点数据支持高并发分页查询。

### 业务上下文

关注和取关属于高频社交动作。它的核心问题不是 SQL 能不能写，而是：

1. 同一个用户可能短时间多次点击。
2. 关注列表和粉丝列表读写都很多。
3. 关注/取关顺序不能乱，否则最终状态会错。

### 实际请求链路

#### 关注

1. `RelationServiceImpl#follow()` 先校验不能关注自己。
2. 再远程查目标用户是否存在。
3. 构建“当前用户关注列表”的 Redis ZSET key。
4. 用 Lua 脚本 `follow_check_and_add.lua` 做“存在性判断 + 写入”原子操作。
5. 如果 ZSET 不存在，就查 DB 全量回填，再重新执行 Lua。
6. 最后发送顺序 MQ，`hashKey` 使用当前用户 ID，保证同一用户的 follow/unfollow 顺序一致。

#### 取关

1. 逻辑和关注对称。
2. 先用 Lua 做原子删除判断。
3. 如果缓存缺失，先从 DB 回填关注列表再删。
4. 再发顺序 MQ 异步落库。

#### 查询关注/粉丝列表

1. 优先从 Redis ZSET 分页查。
2. 如果缓存没有，再查 DB。
3. 查询结果再异步回填 Redis。
4. 粉丝列表和关注列表回填策略不同：
   - 关注列表回填最多 1000 条
   - 粉丝列表回填最多 5000 条

### 设计动机

1. ZSET 用时间戳做 score，天然适合按关注时间倒序分页。
2. Lua 把“判断 + 增删”合并成一个 Redis 原子操作，减少网络往返。
3. 顺序 MQ 确保同一用户多次关注/取关操作按顺序消费，避免最终状态错乱。
4. 列表热点数据缓存到 Redis，可以把分页查询压力从 DB 挪走。

### 技术原理补充

#### 原理说明

1. Lua 脚本在 Redis 单线程模型里执行，所以多个命令可以作为一个原子单元完成。
2. 顺序消息的关键是同一 `hashKey` 落到同一顺序队列并按发送顺序消费。
3. ZSET 的 score 可以天然表示关注时间，所以倒序分页场景非常贴合。
4. 联合唯一索引是数据库层幂等约束，能防止重复关系记录落库。

#### 原理类可能提问

1. 为什么 Lua 能保证原子性，而应用层多次 Redis 调用不行？
   答法：因为 Lua 在 Redis 服务端一次执行完成，不会被其他命令打断；而应用层多次调用中间存在网络和并发窗口，容易出现竞态。
2. 顺序消息能解决所有乱序问题吗？
   答法：不能。它通常只能保证同一队列、同一 hashKey 维度的顺序，不能保证全局所有消息绝对有序。
3. 为什么关注列表更适合 ZSET，而不是 List？
   答法：因为关注列表核心诉求是按关注时间倒序分页，ZSET 天然支持 score 排序；List 更适合简单顺序流，不适合这种时间维度分页。
4. 数据库唯一索引为什么能帮助实现消费幂等？
   答法：因为即使消息重复消费，重复插入同一条关系时数据库会拦住冲突，避免写出多条重复记录，这就是数据库层的最后一道幂等保护。

### 高频追问

#### Q1：为什么这里一定要顺序消息？

答法：
因为同一个用户可能连续做 follow -> unfollow -> follow。如果消息乱序消费，数据库最终状态就可能错误。用 `asyncSendOrderly`，并且用 `userId` 作为 hashKey，可以保证同一用户的操作落在同一消息队列顺序处理。

#### Q2：为什么关注列表和粉丝列表都用 ZSET？

答法：
因为这两类列表都需要按时间排序分页。ZSET 的 member 存用户 ID，score 存关注时间戳，分页和倒序都比较自然。

#### Q3：Lua 在这里的价值是什么？

答法：
如果不用 Lua，应用层要先查 Redis 再决定写 Redis，会有竞态窗口，而且多一次网络往返。Lua 脚本把判断和更新放在 Redis 服务端一次完成，既原子又更快。

### 谨慎表述点

1. “消费者使用联合唯一索引保证幂等性”这件事，源码能看出有顺序消息和异步落库，但联合唯一索引最好结合 DDL 一起说。如果面试时拿不准，就说“数据库侧通过唯一约束或 upsert 语义保证幂等”更稳。
2. 粉丝列表查询里 `noteTotal`、`fansTotal` 还是 TODO 占位值，不能说这个接口已经完整聚合了计数信息。

## 9. 计数服务

### 简历上怎么说

负责高并发场景下的计数服务，用于统计关注数、粉丝数、点赞数等；通过消费用户行为 MQ，在消费者中用 BufferTrigger 做流量聚合，先写 Redis，再异步削峰落库，以支持高并发写。

### 业务上下文

点赞数、粉丝数、收藏数这种计数如果每次都直接同步更新数据库，会很快把热点行打爆，所以需要把“行为事件”和“计数结果”解耦。

### 实际请求链路

以粉丝数为例：

1. 用户关系服务产生 follow/unfollow 行为消息。
2. 计数服务 `CountFansConsumer` 消费消息。
3. 消费时不直接落库，而是先交给 `BufferTrigger`。
4. BufferTrigger 配置为“最多聚合 1000 条，或 1 秒聚合一次”。
5. 聚合后按目标用户分组，计算最终增量。
6. 如果 Redis 里的用户计数 Hash 已存在，则先把粉丝数增量写入 Redis。
7. 再把聚合后的增量 map 发到 `TOPIC_COUNT_FANS_2_DB`。
8. `CountFans2DBConsumer` 获取令牌后再把增量写入数据库。

### 设计动机

1. 行为和计数解耦，业务线程更轻。
2. BufferTrigger 把零散写合并成批量增量写，减少 DB 压力。
3. Redis 先更新，前台读计数时延更低。
4. DB 异步落库作为持久化结果，最终与 Redis 对齐。

### 技术原理补充

#### 原理说明

1. `BufferTrigger` 的核心是按数量或时间窗口聚合离散消息，把逐条处理变成批处理。
2. Redis Hash 适合计数场景，因为多个计数字段可以挂在同一个 key 下做原子增量更新。
3. `RateLimiter` 本质是令牌桶思想，用来限制单位时间内允许多少落库操作通过。
4. 先 Redis 后 DB 的链路，属于“近实时可见 + 异步持久化”的最终一致性方案。

#### 原理类可能提问

1. BufferTrigger 和逐条消费在系统成本上的差异是什么？
   答法：逐条消费每次都要承担反序列化、事务、SQL 往返等固定成本；BufferTrigger 把这些成本摊到一批消息上，所以整体吞吐更高。
2. 为什么计数类更新更适合增量写，而不是频繁全量重算？
   答法：因为用户行为本来就是点赞加一、取消点赞减一这种增量事件，直接累加最便宜。全量重算更适合作为补偿，不适合放在主链路。
3. 令牌桶和漏桶的区别是什么？
   答法：令牌桶强调“有令牌就放行”，允许一定突发；漏桶更强调匀速流出，突发能力弱。这里用令牌桶是为了既能削峰，又保留一点弹性。
4. 为什么先写 Redis 会更接近实时，但也更需要补偿机制？
   答法：因为用户读一般先读 Redis，所以先改 Redis 体验更实时；但 Redis 和 DB 之间是异步链路，就一定存在短时间偏差，因此必须有补偿对齐。

### 高频追问

#### Q1：为什么要先写 Redis 再落库？

答法：
因为计数最怕热点更新。Redis 更适合承接高频增量写，先把热点压力吃掉，再异步把聚合结果落库，可以显著降低数据库写放大。

#### Q2：BufferTrigger 的价值是什么？

答法：
它相当于一个批量聚合器，把 1 秒内或一定条数内的多条行为消息合并后再处理。比如同一个用户在这一批次里多次变更计数，最终只需要合成一次增量结果。

#### Q3：为什么还要 RateLimiter？

答法：
因为即使经过聚合，落库消费者在高峰时仍然可能瞬间很猛。`CountFans2DBConsumer` 里在真正写 DB 前会先获取令牌，相当于给数据库加了一层削峰保护。

### 谨慎表述点

1. “高可用”更多是架构目标，源码里最能证明的是“异步化、聚合、限流、Redis 优先承压”。
2. 面试时多说机制，少说绝对性能数字。

## 10. 数据对齐服务

### 简历上怎么说

负责数据对齐服务设计与实现，通过补偿方案实现计数最终一致性；使用 XXL-Job 创建日增量计数变更表；消费 MQ 记录当日变更；设计分片广播任务并行执行凌晨对齐。

### 业务上下文

只要系统里有“Redis 先更新、DB 异步落库、消息异步消费”，就必然会有极端场景下的数据偏差。所以计数服务必须搭配一个兜底对齐服务，做最终一致性修正。

### 实际请求链路

1. 白天各业务服务继续产生日志或行为消息。
2. 数据对齐服务消费这些 MQ，把“今天哪些用户/笔记发生了变更”记到临时表里。
3. `CreateTableXxlJob` 会按天、按分片预建增量记录表。
4. 凌晨执行分片广播任务，比如 `FansCountShardingXxlJob`。
5. 每个分片只处理自己那一份数据。
6. 对每个变更对象重新基于事实表做 count。
7. 回写 DB 计数表。
8. 如果 Redis 缓存还在，也同步修正 Redis。
9. 最后调用搜索服务重建相关索引文档。

### 设计动机

1. 正常链路优先性能，允许短时间内最终一致。
2. 对齐服务兜底修复异步链路中的消息丢失、重复消费、局部失败等问题。
3. 分片广播可以把大任务拆小，保证在夜间窗口内完成。

### 技术原理补充

#### 原理说明

1. 数据对齐本质是补偿机制，适用于主链路为了吞吐做了异步化后的最终一致性兜底。
2. Bloom 在这里不是做业务准确判断，而是做“今天这个对象是否已经记录过变更”的去重优化。
3. XXL-Job 分片广播的核心是把大任务拆给多个执行器并行跑，缩短处理窗口。
4. 夜间重算不是为了强一致，而是用时间换吞吐和主链路轻量化。

#### 原理类可能提问

1. 最终一致性和强一致性的区别是什么？
   答法：强一致性要求每次读都立即看到最新写入；最终一致性允许短时间不一致，但系统会在之后收敛到一致状态。
2. 为什么补偿系统通常只处理增量对象，而不做全表重刷？
   答法：因为全表重刷成本太高，而真正发生变化的对象通常只占一部分。只处理增量对象能把补偿代价控制在可接受范围内。
3. 分片任务为什么能缩短任务执行时间？
   答法：因为它把一个大任务拆给多个执行器并行处理，从串行变成并行，整体窗口自然缩短。
4. 为什么补偿类任务通常放在低峰时段？
   答法：因为补偿常常涉及重算和批量更新，比较吃资源。放在低峰时段执行，对主业务流量影响最小。

### 高频追问

#### Q1：为什么不直接在主链路里做强一致？

答法：
因为主链路核心目标是吞吐和响应速度。如果每次用户行为都要同步更新多处存储并保证强一致，复杂度和延迟都会很高。这里选择的是“主链路高性能 + 离线补偿最终一致”。

#### Q2：为什么要预创建临时表？

答法：
因为对齐服务按天存增量变更记录，预创建表可以避免任务正式跑的时候临时建表带来额外抖动，也便于做分片管理和历史清理。

#### Q3：分片广播怎么理解？

答法：
就是一个 XXL-Job 任务在多分片实例上并行执行，每个分片只处理自己的 hash 区间，最后一起完成整批数据校准。

### 谨慎表述点

1. 这个服务的价值很大，但面试时一定要说清楚它是“最终一致性补偿”，不是在线实时强一致。
2. 对齐逻辑里还会触发搜索索引重建，这个点容易加分，别漏讲。

## 11. 搜索服务

### 简历上怎么说

负责笔记、用户索引设计；实现笔记和用户搜索接口，支持中文分词、高亮、条件过滤、自定义评分；结合 Canal 监听 MySQL Binlog 做增量索引构建。

### 业务上下文

搜索不是简单的 `like '%关键词%'`。一旦进入内容平台场景，通常会遇到：

1. 标题、话题、昵称等多字段匹配。
2. 结果要按综合相关性排序。
3. 需要高亮展示命中词。
4. 用户信息和笔记信息变化后，索引要尽快同步。

### 实际请求链路

#### 搜索笔记

1. `NoteSearchServiceImpl#searchNote()` 收到关键字、排序类型、发布时间范围、类型过滤等条件。
2. 构建 ES `multi_match` 查询，重点匹配标题和话题，其中标题权重更高。
3. 如果用户选了类型或发布时间范围，再加 filter。
4. 如果选了“最新/最多点赞/最多评论/最多收藏”，就走对应 field sort。
5. 如果没选特定排序，就走 `function_score` 自定义评分，把点赞、收藏、评论纳入综合分。
6. 再加 title 高亮。
7. 执行查询，组装返回值。

#### 增量索引

1. `CanalClient` 建立 Canal 连接并订阅库表变更。
2. `CanalSchedule` 每 100ms 拉一次批量消息。
3. 解析出表名、事件类型和列值。
4. 如果是 `t_note`，根据状态和可见性决定“重建索引还是删文档”。
5. 如果是 `t_user`，根据状态和逻辑删除标记决定“重建用户索引，并联动重建该用户笔记索引”或“删文档”。

### 设计动机

1. 搜索索引和业务库解耦。
2. 查询时用 ES 承接复杂文本检索。
3. 写入时用 Canal 捕获 Binlog，减少业务代码里到处手写“更新索引”的侵入。
4. 综合排序不是只看文本匹配，也把互动量纳入评分，更符合内容平台语义。

### 技术原理补充

#### 原理说明

1. Elasticsearch 的核心是倒排索引，适合全文检索，不适合承担事务型主存储。
2. `function_score` 的思想是“先有文本相关度，再叠加业务分值”，让结果更贴近平台业务。
3. 高亮本质上是搜索命中后，ES 返回匹配片段信息，再对命中字段做标签包装。
4. Canal 属于 CDC，即 Change Data Capture，通过监听 Binlog 感知数据库变化。

#### 原理类可能提问

1. 什么是倒排索引，为什么 MySQL Like 不适合复杂搜索？
   答法：倒排索引是“词 -> 文档列表”的结构，适合快速找到包含某词的文档。MySQL Like 往往更像逐行扫描，面对复杂全文搜索和高并发时成本很高。
2. `function_score` 为什么适合内容平台排序？
   答法：因为内容平台排序不能只看关键词匹配，还要看互动质量。`function_score` 能把文本相关度和点赞、评论、收藏这些业务信号合起来。
3. 高亮是搜索引擎做的还是应用层自己拼的？
   答法：主要是搜索引擎做的。ES 会返回命中字段的高亮片段，应用层更多是把它装配进返回结果。
4. CDC 和业务双写相比，优缺点分别是什么？
   答法：CDC 的优点是低侵入、以数据库为事实源；缺点是链路更长、排障更复杂。业务双写实时性更直接，但侵入主业务且容易漏写或耦合过重。

### 高频追问

#### Q1：为什么综合排序要做 function_score？

答法：
因为内容搜索不能只看文本命中，互动数据也会影响结果质量。比如标题匹配相近的两篇笔记，点赞、收藏、评论更高的通常更值得排前面，所以我把这些字段通过 `field_value_factor` 叠加到评分里。

#### Q2：为什么用户更新后要重建笔记索引？

答法：
因为笔记索引里通常冗余了作者昵称、头像等字段。用户资料一变，不只是用户索引要更新，对应笔记索引里冗余的作者信息也要刷新。

#### Q3：Canal 的优势是什么？

答法：
它直接订阅 Binlog，业务服务不用在每次改库时都显式调用搜索服务更新索引，侵入性更低，也更接近真实生产里的增量索引同步方式。

### 谨慎表述点

1. 中文分词、高亮、过滤、自定义评分，这些源码都能支撑。
2. “Logstash 做全量索引构建”在仓库里没有直接看到 logstash 配置文件，面试时可以说“全量构建是有这个方案/过程”，但不要说代码仓里完整落了 logstash pipeline。

## 12. 评论服务

### 简历上怎么说

负责评论服务设计与实现；通过冗余字段优化查询；评论发布异步化；MQ 发送失败重试；消费端批量写入；分页查询结合 Redis + 本地缓存；评论点赞/取消点赞通过 MQ 异步处理、批量消费、批量写库和内存合并提升性能。

### 业务上下文

评论链路的难点比笔记详情更多，因为它同时有：

1. 一级评论、二级评论结构。
2. 评论正文和元数据分库存储。
3. 评论热度排序。
4. 评论点赞高并发写。
5. 评论总数、子评论数、最早回复评论这些冗余字段需求。

### 实际请求链路

#### 发布评论

1. `CommentServiceImpl#publishComment()` 校验评论正文和图片不能同时为空。
2. 通过分布式 ID 服务生成评论 ID。
3. 组装 `PublishCommentMqDTO`。
4. 使用 `SendMqRetryHelper.asyncSend()` 发 MQ。
5. 消费端 `Comment2DBConsumer` 批量消费评论发布消息。
6. 先批量查被回复评论，识别一级/二级评论结构。
7. DTO 转 BO，补齐 `level`、`parentId`、`replyUserId`、`contentUuid` 等字段。
8. 在事务里先批量写评论元数据，再批量写评论正文到 KV。
9. 成功后发送计数 MQ，并同步热点一级评论到 Redis ZSET。

#### 一级评论分页

1. 先从 Redis Hash 里取评论总数。
2. 再看评论 ZSET 缓存是否存在。
3. 热门页优先从 Redis ZSET 取评论 ID。
4. 详情先查本地 Caffeine。
5. 本地没命中再批量查 Redis String。
6. Redis 再没命中才批量查 DB。
7. 最后把详情再同步回 Redis 和本地缓存。

#### 评论点赞

1. 逻辑和笔记点赞类似，先用布隆过滤器做快速判断。
2. 再用 DB 或其他缓存做兜底确认。
3. 行为消息发到 MQ。
4. `LikeUnlikeComment2DBConsumer` 批量消费。
5. 按 commentId 分组，再对同一用户的多次操作只保留最后一次。
6. 点赞批量插入，取消点赞批量删除。

### 设计动机

1. 评论发布走异步，是因为评论写入往往会连带更新计数、热度、缓存，如果全同步会拖慢接口。
2. 评论正文拆到 KV，可以减轻评论元数据表压力。
3. 一级评论分页做“本地缓存 + Redis + DB”三级查找，是为了把热点评论读取做快。
4. 评论点赞消费端做“内存合并 + 批量 SQL”，是为了减少频繁写库。

### 技术原理补充

#### 原理说明

1. 评论发布走 MQ，本质是把同步事务拆成“前台接单、后台处理”的异步削峰模型。
2. 多级缓存的目标是逐层拦截热点请求，让最贵的 DB 查询尽量少发生。
3. 评论点赞的 Bloom 防重和笔记点赞一样，核心是用概率结构减少数据库判重次数。
4. 批量消费和批量写库的本质是减少事务次数、连接切换和 SQL 往返。

#### 原理类可能提问

1. 异步发布评论为什么能降低接口 RT？
   答法：因为前台请求只需要把“发布请求成功接住”这件事先完成，后面的正文写入、热度更新、计数联动都交给后台处理，主线程等待更少。
2. 批量消费为什么通常比逐条消费更省数据库资源？
   答法：因为它把很多小事务合并成少量大事务，减少连接切换、SQL 次数和事务提交次数。
3. 为什么一级评论和二级评论的缓存模型不完全一样？
   答法：因为一级评论更偏公共热点，访问量高，适合更积极缓存；二级评论更分散、更依赖父评论上下文，所以缓存策略通常更保守。
4. 为什么删除评论后不仅要删 Redis，还要删本地缓存？
   答法：因为 Redis 只解决共享缓存失效，本地 Caffeine 是每个节点自己的一份。如果不删本地缓存，其他节点仍可能继续返回旧数据。

### 高频追问

#### Q1：为什么评论模型要冗余 `replyTotal`、`replyCommentId` 这类字段？

答法：
因为评论展示经常需要“这条一级评论有多少回复”“最先展示哪条回复”这种信息。如果每次都实时 join 或子查询，会很重，所以我在模型里保留了冗余字段，用空间换查询效率。

#### Q2：为什么评论发布不直接写数据库？

答法：
因为评论发布后不只是写一条评论，还会涉及正文存储、计数更新、热点评论缓存、热度相关逻辑。拆成 MQ 异步链路以后，前台接口更快，后续扩展也更容易。

#### Q3：评论点赞消费为什么还要做“只保留最后一次操作”？

答法：
因为用户可能短时间连续点“赞/取消赞/赞”。如果每条消息都直接落库，会产生大量无效写。合并后只保留最终状态，数据库压力会小很多。

### 谨慎表述点

1. `SendMqRetryHelper` 里“重试”是真实落地的，但“多次重试失败后的兜底写库/异步失败消息机制”目前还是 `TODO`，没有完整落地。

   当前已实现 Spring Retry + 异步重试，兜底写失败消息表是设计预留，源码里还没补完。

## 附录：跨模块技术原理高频问答

这一部分不是按服务拆，而是按中间件和通用技术原理拆。很多面试官不会顺着你的服务边界问，而是会顺着“你提到的技术名词”继续追问。

### 1. Redis 为什么适合做缓存？

答法：
Redis 是内存数据库，单次读写延迟低，支持丰富数据结构，非常适合承接热点读和热点增量写。在这个项目里，Redis 主要承担缓存、短期状态、计数增量和关系集合这几类职责。

继续追问时可以补：

1. Redis 快的核心原因是内存访问快、数据结构简单、避免了复杂磁盘 IO。
2. 单线程不等于低性能，因为很多缓存场景瓶颈在网络和内存，不在 CPU。

### 2. Redis 常见数据结构为什么这样选？

答法：

1. `String` 适合简单 KV、JSON 串和验证码。
2. `Hash` 适合计数字段，比如用户计数、评论计数。
3. `ZSET` 适合带排序语义的数据，比如点赞时间、关注时间、评论热度。
4. Bloom 适合高并发判重场景。

如果被问“为什么点赞列表不用 List”，答：
因为 List 更适合顺序消费，不适合按 score 排序分页；点赞列表更需要时间排序和判存在，ZSET 更合适。

### 3. 什么是缓存穿透、击穿、雪崩？

答法：

1. 穿透：请求的数据根本不存在，每次都打数据库。
2. 击穿：某个热点 key 失效瞬间，大量请求同时打数据库。
3. 雪崩：大量 key 在同一时间集中失效，数据库整体压力暴涨。

你这个项目里的对应策略：

1. 穿透：空值缓存。
2. 雪崩：随机 TTL。
3. 击穿：二级缓存和异步回填缓解，但没有做到基于锁的强单飞控制。

### 4. Bloom Filter 的原理是什么？

答法：
Bloom Filter 本质是一个位数组加多个哈希函数。插入元素时，把多个哈希位置都置成 1；查询元素时，只要有一个位置是 0，就说明它一定不存在；如果所有位置都是 1，只能说明它可能存在。

重点结论：

1. 不存在一定不存在。
2. 存在只是“可能存在”。
3. 它会误判，不会漏判。

### 5. 为什么 Bloom 会误判但不会漏判？

答法：
真实存在的元素在插入时一定会把对应 bit 置成 1，所以查询时不可能发现 0，因此不会漏判；但不同元素可能碰巧把同一批 bit 都置成 1，所以一个没插入过的元素也可能被判断为“存在”，这就是误判。

### 6. ZSET 的原理和适用场景是什么？

答法：
ZSET 是有序集合，member 唯一，score 可排序，非常适合“对象 + 时间戳/热度值”模型。

在你的项目里：

1. 关注列表的 score 是关注时间。
2. 点赞/收藏列表的 score 是操作时间。
3. 热点评论的 score 是热度值。

### 7. Lua 脚本为什么能保证原子性？

答法：
Redis 执行 Lua 脚本时，会把整段脚本作为一个不可分割的命令在单线程里执行，在执行结束前不会被其他命令插入。所以判断和更新可以在 Redis 服务端一次完成。

### 8. MQ 为什么能提升系统吞吐？

答法：
MQ 的核心价值是削峰和解耦。原本一个请求要同步完成多件事，现在可以拆成：

1. 主链路先完成最关键动作。
2. 计数、索引、热度、缓存失效等动作异步交给消费者。

这样做的收益是：

1. 接口 RT 更低。
2. 下游故障不会立刻拖死主流程。
3. 可以通过批量消费和限流更柔和地打数据库。

### 9. 顺序消息的意义是什么？

答法：
顺序消息适合那种“同一业务主键下，多次操作先后顺序会影响最终状态”的场景，比如关注/取关、点赞/取消点赞。它不能保证全局顺序，但能尽量保证同一 `hashKey` 维度下的顺序。

### 10. 批量消费为什么通常比逐条消费更高效？

答法：
因为逐条消费会重复产生 JSON 反序列化、事务切换、数据库连接切换和 SQL 往返开销。批量消费把这些固定成本摊薄了，所以吞吐更高。

### 11. RateLimiter 的令牌桶原理是什么？

答法：
系统会按固定速率往桶里放令牌，请求必须先拿到令牌才能继续。拿不到就等待或失败。这样就能把流量限制在系统可承受范围内。

在你的项目里，RateLimiter 主要用在消费者落库前，目标是保护数据库，而不是限前台请求。

### 12. CompletableFuture 为什么能降低接口耗时？

答法：
如果两个下游调用互不依赖，串行执行总耗时约等于 `A + B`，并行执行总耗时更接近 `max(A, B)`。所以在笔记详情里“查作者信息 + 查正文内容”很适合并发拼装。

### 13. Caffeine 适合解决什么问题？

答法：
Caffeine 是本地缓存，优点是快，缺点是不共享。所以它适合极热点读和单节点内高频重复访问，不适合作为唯一缓存层。因此你的项目才会用“Caffeine + Redis”两级缓存。

### 14. Canal 的原理是什么？

答法：
Canal 本质是在模拟 MySQL 从库协议，订阅主库 Binlog，把数据库变化转成可消费的变更事件。这样业务服务不需要在每次写库时显式调用搜索服务更新索引。

### 15. 什么是倒排索引？

答法：
倒排索引不是“文档对应词”，而是“词对应文档列表”。比如搜索词 `咖啡`，索引里直接记录哪些文档包含这个词，所以全文搜索时不需要扫描所有文档。

### 16. function_score 为什么适合内容平台？

答法：
内容平台排序不能只看文本匹配，因为用户更关心“相关且优质”的内容。`function_score` 可以在文本相关度基础上，再叠加点赞、评论、收藏等业务信号，让排序更符合平台目标。

### 17. Cassandra 和 MySQL 的核心差异怎么讲？

答法：

1. MySQL 强在事务、一致性、结构化查询。
2. Cassandra 强在按主键高吞吐读写和横向扩展。

所以项目里把用户、笔记、关系、计数元数据放 MySQL，把正文类 KV 数据放 Cassandra，本质是按访问模式做选型。

### 18. 什么是最终一致性？

答法：
最终一致性不是说系统任意时刻都一致，而是说在没有新更新持续进入的情况下，系统经过一段时间后会收敛到一致状态。

在这个项目里：

1. Redis 计数可能先变。
2. DB 计数稍后写入。
3. 数据对齐服务夜间重算补偿。

最后三者收敛，这就是最终一致性。

### 19. 为什么高并发架构里几乎都需要补偿系统？

答法：
因为只要主链路用了异步化、MQ、缓存优先、批量聚合，就一定会引入少量延迟和异常路径。补偿系统不是替代主链路，而是兜住极端情况下的数据偏差。

1. 先按服务模块讲业务链路。
2. 面试官开始追问中间件原理时，再切到这一页。
3. 每个原理问题先给一句结论，再补 2 到 3 句结合项目的落地解释。

这样你不会答成纯八股，也不会答成只会背项目流程。
