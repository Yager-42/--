# 小哈书项目面试拷问稿

> 说明：
> 1. 本文档基于 `xiaohashu` 当前仓库代码、SQL 和你的简历条目整理。
> 2. 目标不是润色简历，而是把每个点讲到“能解释上下文、主链路、关键设计、为什么这样做、边界问题”。
> 3. 文中单独标了“谨慎表述”，这些地方面试时不要说过头。

## 一、先把项目整体讲清楚

### 1. 项目定位

这个项目本质上是一个内容社区后端，核心对象有用户、笔记、评论、关注关系、点赞/收藏/计数、搜索索引、对象存储、短文本存储。

它不是单体 CRUD，而是围绕两个核心目标设计的：

- 第一，微服务拆分，把认证、用户、内容、关系、搜索、存储、计数等职责拆开，降低耦合。
- 第二，高并发场景下把“强实时写”改造成“前台快速响应，后台异步落库/聚合/补偿”的模式。

### 2. 整体架构主线

你可以把系统理解成四层：

- 接入层：`gateway`
- 身份层：`auth`
- 核心业务层：`user`、`note`、`comment`、`user-relation`
- 基础能力层：`distributed-id-generator`、`oss`、`kv`、`count`、`data-align`、`search`

### 3. 一条典型业务链路

以“用户登录后发布笔记，再被其他用户点赞、评论、搜索到”为例：

1. 用户先通过 `auth` 服务登录，拿到 Sa-Token 令牌。
2. 请求经过 `gateway`，网关统一做登录校验和角色校验。
3. 笔记发布时，`note` 服务调用 `distributed-id-generator` 生成笔记 ID，调用 `kv` 存正文，调用 `user` 获取发布者信息，必要时调用 `oss` 处理图片或视频地址。
4. 笔记详情读取时，`note` 先查本地缓存，再查 Redis，再查 DB；如果正文不为空，再并发调用 `kv`；同时并发调用 `user` 查询作者信息。
5. 点赞、收藏、关注、评论这些高频写操作先写 Redis / 发 MQ，异步批量落库。
6. `count` 服务消费行为 MQ，先聚合更新 Redis 计数，再限流写 MySQL。
7. `data-align` 服务夜间基于日增量记录做补偿对齐，修正最终计数。
8. `search` 服务通过 Canal 监听 MySQL Binlog，把用户和笔记的变化同步到 Elasticsearch，支持搜索和高亮。

### 4. 面试开场建议

先别上来就讲中间件。推荐顺序是：

1. 先讲项目是什么业务。
2. 再讲我负责哪些服务。
3. 然后讲项目为什么需要这么拆。
4. 最后再挑 2 到 3 个高并发场景深讲。

一句话版本可以这么说：

“这个项目是一个内容社区后端，我主要负责网关、认证、对象存储、分布式 ID、用户、笔记、用户关系、计数、数据对齐、评论和搜索这些模块。整体设计重点是把高并发写请求先抗在 Redis 和 MQ 上，再通过计数聚合、缓存失效广播、夜间对齐和 ES 增量索引，保证读性能和最终一致性。”

## 二、网关服务 Gateway

### 1. 业务上下文

所有外部请求都会先经过网关。网关的职责不是写业务，而是统一做三件事：

- 路由转发
- 登录校验和角色权限校验
- 把用户身份透传给下游服务

如果没有网关，每个服务都要重复做鉴权和上下文解析，代码会散、维护成本高。

### 2. 请求链路

1. 请求进入网关。
2. `SaTokenConfigure` 中注册的 `SaReactorFilter` 先执行。
3. 它会对除登录、验证码发送外的接口统一执行 `StpUtil.checkLogin()`。
4. 对特定接口继续做角色校验，比如 `/auth/logout` 需要 `common_user` 角色。
5. 理论上通过 `AddUserId2HeaderFilter` 从 Token 中取出 `loginId`，把用户 ID 放进请求头。
6. 下游服务再通过 `HeaderUserId2ContextFilter` 把请求头中的用户 ID 放入 `ThreadLocal`。
7. 如果下游又调用别的服务，`FeignRequestInterceptor` 再把用户 ID 继续透传出去。

### 3. 为什么这样设计

- 鉴权统一放网关，避免每个微服务重复写一遍。
- 用户 ID 通过 Header 透传，避免下游服务重复解析 Token。
- 下游服务只依赖统一上下文，业务代码更干净。

### 4. 你要能答出来的追问

#### 为什么不用每个服务自己解析 JWT？

因为那样每个服务都要重复做 Token 校验、异常处理和用户上下文提取。统一放网关可以收敛逻辑，也方便以后扩展黑名单、限流、审计。

#### 网关如何知道用户角色和权限？

`StpInterfaceImpl` 会根据用户 ID 去 Redis 读角色列表和角色对应的权限列表。也就是说，角色和权限数据不是每次临时查库，而是提前由用户服务放到 Redis 中。

#### 用户 ID 透传到下游后怎么用？

下游服务通过过滤器把 Header 中的用户 ID 放入 `LoginUserContextHolder`。这样业务层直接 `getUserId()` 就能拿到当前登录人。

### 5. 谨慎表述

- 简历里“全局过滤器解析 Token 并透传用户 ID”这个思路是对的。
- 但当前代码里 `AddUserId2HeaderFilter` 构造了 `newExchange`，最终返回时却仍然 `chain.filter(exchange)`，没有把新请求对象传下去。
- 所以面试时更稳妥的说法是：
  - “我设计的是在网关统一鉴权后，把用户 ID 透传给下游；下游配套了 Header -> ThreadLocal -> Feign 透传链路。”
  - 不要说成“我已经在线上验证这个 Header 透传代码完全没问题”，除非你自己跑过并改过这个点。

## 三、认证服务 Auth

### 1. 业务上下文

认证服务解决的是“用户怎么进系统”的问题，不是“用户资料怎么维护”的问题。

它负责：

- 账号密码登录
- 手机验证码登录
- 用户自注册
- 登出
- 修改密码
- 短信验证码发送和防刷

### 2. 登录与注册链路

#### 验证码登录链路

1. 前端传手机号和验证码到 `AuthServiceImpl.loginAndRegister()`。
2. 服务先根据登录类型判断走验证码登录。
3. 用手机号拼 Redis Key，读取之前发送的验证码。
4. 对比用户提交的验证码和 Redis 中的验证码。
5. 校验通过后，调用用户服务 `registerUser(phone)`。
6. 这里的注册接口是“幂等注册”：如果用户已存在，就直接返回用户 ID；不存在就创建。
7. 拿到用户 ID 后调用 `StpUtil.login(userId)` 完成登录。
8. 再通过 `StpUtil.getTokenInfo()` 返回 Token。

#### 密码登录链路

1. 根据手机号调用用户服务查询用户。
2. 取回密文密码。
3. 使用 `BCryptPasswordEncoder.matches()` 做密码比对。
4. 成功后调用 Sa-Token 登录并返回 Token。

### 3. 验证码发送链路

1. `VerificationCodeServiceImpl.send()` 接收手机号。
2. 先判断 Redis 中该手机号对应验证码 Key 是否还存在。
3. 如果存在，说明上一次验证码还没过期，直接拒绝，起到防刷作用。
4. 如果不存在，则生成 6 位随机验证码。
5. 然后用自定义线程池异步调用阿里云 SMS 发送短信。
6. 同时把验证码写入 Redis，过期时间 3 分钟。

### 4. 为什么这样设计

- 登录和注册放在一起，是为了减少验证码登录时“先判断是否注册，再决定登录或注册”的前后端交互。
- 验证码先写 Redis，再异步发短信，接口响应更快。
- 发送频率通过 Redis Key 是否存在来控制，成本低、实现直接。
- 密码不自己做加盐逻辑，而是直接使用 BCrypt，让框架负责随机盐和哈希。

### 5. 你要能答出来的追问

#### 为什么说是 RBAC？

因为认证本身只解决“我是谁”，权限判断是在网关侧基于用户角色和角色权限做的，这就是典型的 RBAC 思路：用户 -> 角色 -> 权限。

#### 为什么验证码登录还能叫自注册？

因为用户第一次用手机号验证码登录时，认证服务会调用用户服务完成幂等注册；如果已经注册，直接返回已有用户 ID。

#### BCrypt 为什么安全？

因为 BCrypt 自带随机盐，而且是慢哈希，能提高暴力破解成本。代码中 `PasswordEncoderConfig` 就是 `new BCryptPasswordEncoder()`。

### 6. 谨慎表述

- “短信接口防刷”当前代码里主要体现为“验证码未过期不允许重复发送”，这是基础版防刷，不是严格意义上的多维限流。
- 所以面试时可以说“实现了基于 Redis 的基础防刷控制”，不要夸成完整风控系统。

## 四、分布式 ID 生成服务

### 1. 业务上下文

多个微服务都要生成业务 ID，如果每个服务自己用数据库自增主键，会遇到：

- 分库分表困难
- 多服务之间 ID 规则不统一
- 很难保证全局唯一和高吞吐

所以项目单独抽了一个分布式 ID 服务。

### 2. 关键链路

1. 业务服务通过 Feign 调用 ID 服务。
2. 如果要业务号段 ID，走 `LeafController.getSegmentId()`。
3. 如果要雪花 ID，走 `LeafController.getSnowflakeId()`。
4. 号段模式底层用 `SegmentIDGenImpl`，依赖 MySQL 的号段表批量预取。
5. 雪花模式底层用 `SnowflakeIDGenImpl`，依赖 ZooKeeper 分配 workerId。

### 3. 为什么这样设计

- 号段模式适合吞吐要求高的业务 ID。
- 雪花模式适合快速生成全局唯一 long 型 ID。
- 把两种模式都做出来，是为了让不同业务按需求选型。

### 4. 你要能答出来的追问

#### 号段模式为什么快？

因为不是每次都打数据库。它一次从数据库取一大段号，缓存到内存里，后续发号都在本地完成，只有号段快用完时才去 DB 续下一段。

#### 雪花模式最大风险是什么？

时钟回拨。因为雪花 ID 依赖时间戳，如果机器时间回退，就可能出现重复或乱序风险。Leaf 的雪花实现里对这类场景做了异常处理。

#### 为什么还需要 ZooKeeper？

因为雪花算法需要 workerId，多节点部署时必须有一个协调机制保证 workerId 不冲突，当前实现就是通过 ZooKeeper 分配和维护。

### 5. 谨慎表述

- “整合 Leaf 源码、提供号段和雪花两种模式”可以放心讲。
- “JMeter 吞吐 22000+/s、平均 4ms”当前仓库没有压测报告，面试时要讲成“我自己压测得到的数据”，不要说“仓库内有证据可以直接证明”。

## 五、OSS 对象存储服务

### 1. 业务上下文

图片、视频、头像、背景图这些非结构化数据不适合直接放数据库，所以单独抽对象存储服务。

### 2. 上传链路

1. 上游业务把文件发给 `FileController.uploadFile()`。
2. `FileServiceImpl.uploadFile()` 处理业务层逻辑。
3. 工厂 `FileStrategyFactory.getFileStrategy()` 根据配置返回当前存储策略。
4. 如果配置是 `aliyun`，走 `AliyunOSSFileStrategy`。
5. 如果配置是 `minio`，走 `MinioFileStrategy`。
6. 策略内部负责生成对象名、上传文件、返回 URL。

### 3. 为什么这样设计

- 业务层只依赖统一接口 `FileStrategy`，不关心底层是阿里云还是 MinIO。
- 工厂 + 策略模式方便扩展，后续再加腾讯云 COS 也只需要补一套策略实现。
- 通过配置切换当前策略，避免业务代码改动。

### 4. 你要能答出来的追问

#### 为什么不是 if-else 直接写死？

因为对象存储是明显的可替换能力，适合用策略模式封装差异。业务层只看统一的 `uploadFile()`，底层差异下沉到策略实现里。

#### 文件名为什么要改成随机对象名？

避免重名覆盖，也降低暴露原始文件名带来的信息泄露风险。

### 5. 谨慎表述

- “支持阿里云 OSS 和 Minio”成立。
- 但“通过 Nacos 动态注册策略 Bean”说法不准确。
- 更准确地说是：
  - “通过配置中心下发存储类型，配合 `@RefreshScope` 和工厂模式动态切换当前生效的存储策略。”

## 六、KV 短文本键值存储服务

### 1. 业务上下文

笔记正文和评论正文虽然也是业务数据，但它们有两个特点：

- 内容通常是短文本或中等长度文本
- 结构化查询诉求弱，更多是按主键取值

所以项目没有把正文和元数据混在一张 MySQL 表里，而是拆到 KV 服务中存储。

### 2. 关键链路

#### 笔记正文

1. `note` 服务发布笔记时生成 `contentUuid`。
2. 调用 `keyValueRpcService.saveNoteContent(contentUuid, content)`。
3. `kv` 服务 `NoteContentServiceImpl.addNoteContent()` 把内容写入 Cassandra。
4. 查询笔记详情时，再通过 `findNoteContent(uuid)` 查回正文。

#### 评论正文

1. 评论服务批量消费评论发布 MQ 后，先落评论元数据，再调用 `keyValueRpcService.batchSaveCommentContent()`。
2. 查询评论列表时，再通过 `batchFindCommentContent(noteId, yearMonth, contentId)` 批量取回正文。

### 3. 为什么这样设计

- 正文和元数据分离，主业务表更轻，结构化查询更稳定。
- 评论列表是批量读取场景，KV 服务提供批量接口，避免 N 次 RPC 和 N 次 DB 查询。
- 评论正文的复合主键按业务维度分桶，降低单分区热点风险。

### 4. 你要能答出来的追问

#### 评论正文为什么用 `noteId + yearMonth + contentId` 做复合主键？

这是为了兼顾定位能力和分区均衡。既能按业务维度批量查，又不至于把所有内容打进单一分区。

#### 为什么不直接放 MySQL？

因为正文不是高关联结构化字段，拆出去后主业务库可以更聚焦索引和条件查询。

### 5. 谨慎表述

- 你可以说“基于 Cassandra 做短文本 KV 存储”。
- 不要说“基于 RocksDB 存储引擎上构建的 Cassandra”，当前仓库没有 RocksDB 代码，这句话和仓库不一致。

## 七、用户服务

### 1. 业务上下文

用户服务负责的是用户资料和用户基础身份数据，不负责认证流程本身。

它主要做：

- 用户注册
- 根据手机号查用户
- 根据 ID 查用户
- 修改密码
- 修改头像、背景图、昵称、简介等资料
- 维护用户角色信息

### 2. 注册链路

1. 认证服务通过手机号调用用户服务注册。
2. 用户服务先按手机号查库，已注册就直接返回已有用户 ID。
3. 未注册则调用分布式 ID 服务生成用户 ID 和小哈书号。
4. 插入 `t_user`。
5. 再插入 `t_user_role`，给用户分配默认角色。
6. 最后把用户角色写入 Redis，供网关鉴权使用。

### 3. 用户信息查询链路

1. 先查本地 Caffeine 缓存。
2. 本地没命中再查 Redis。
3. Redis 没命中再查 MySQL。
4. 如果数据库没有该用户，会把 `"null"` 作为空值写入 Redis，并设置短过期时间，防止缓存穿透。
5. 查到后再异步回填 Redis 和本地缓存。

### 4. 为什么这样设计

- 本地缓存解决单实例热点。
- Redis 解决多实例共享热点。
- 空值缓存解决穿透。
- 过期时间带随机值，降低同一时间大量缓存失效导致的雪崩风险。

### 5. 你要能答出来的追问

#### 为什么要两级缓存？

因为纯 Redis 每次都要走网络，本地缓存命中时更快；但本地缓存又不能共享，所以还需要 Redis 做分布式层面的兜底。

#### BCrypt 的随机盐体现在哪里？

不是我手写盐值，而是 `BCryptPasswordEncoder` 内部在每次 `encode()` 时自动生成随机盐。

### 6. 谨慎表述

- “Redis + Caffeine 二级缓存、防雪崩、防穿透”能对上代码。
- “防击穿”这里主要是通过双层缓存和异步回填缓解，但没有看到显式分布式锁，面试时不要说成“我用互斥锁彻底解决缓存击穿”。

## 八、笔记服务

### 1. 业务上下文

笔记服务是内容中心，负责：

- 发布笔记
- 编辑笔记
- 查询笔记详情
- 删除笔记
- 置顶
- 修改可见性
- 点赞、取消点赞
- 收藏、取消收藏

### 2. 发布链路

1. 校验笔记类型是图文还是视频。
2. 图文校验图片数量，视频校验视频地址。
3. 调分布式 ID 服务生成笔记 ID。
4. 如果正文不为空，先生成 `contentUuid`，再调用 KV 服务存正文。
5. 组装 `NoteDO` 入库。
6. 发布成功后发 MQ，通知计数和搜索等下游。

### 3. 查询详情链路

1. 先查本地 Caffeine。
2. 本地没命中查 Redis。
3. Redis 没命中查 MySQL。
4. 如果笔记存在，再并发调两个下游：
   - 调用户服务拿作者信息
   - 调 KV 服务拿正文内容
5. 用 `CompletableFuture.allOf()` 等两个结果都回来后组装详情 VO。
6. 最后异步回填 Redis 和本地缓存。

### 4. 点赞链路

1. 先校验笔记是否存在，顺便拿到作者 ID。
2. 根据当前用户 ID 构造点赞布隆过滤器 Key。
3. 用 Lua + Bloom 判断是否“可能点过赞”。
4. 如果 Bloom 不存在，就查 DB 初始化 Bloom，并做第一次兜底。
5. 然后更新用户维度的点赞 ZSET，score 是点赞时间戳。
6. 最后发 MQ 异步落库。
7. 消费者侧结合 `RateLimiter` 做削峰，防止打垮 DB。

### 5. 为什么这样设计

- 笔记详情查询是读多写少场景，适合两级缓存。
- 作者信息和正文内容来自两个下游服务，并发调用能降低总耗时。
- 点赞写入量大，用 Bloom + ZSET + MQ 可以把热点抗在 Redis 和异步队列层。
- 笔记更新、删除后通过 MQ 广播删本地缓存，解决多节点本地缓存一致性问题。

### 6. 你要能答出来的追问

#### 为什么笔记详情要用 `CompletableFuture`？

因为作者信息和正文内容彼此独立，串行调用总耗时是两段 RPC 累加；并行后总耗时接近较慢的那个请求。

#### 为什么点赞既要 Bloom 又要 ZSET？

Bloom 用来快速做“可能存在”判断，适合高并发防重；ZSET 用来记录用户最近点赞列表，并支持时间维度的排序和后续查询。

#### 为什么更新笔记不直接删本地缓存就够了？

因为服务是集群部署，每个节点都有自己的 Caffeine，本节点删掉不代表其他节点也删掉，所以要发广播消息通知所有节点失效。

### 7. 谨慎表述

- “CompletableFuture 并发调用下游、二级缓存、广播删本地缓存、Redis + MQ 抗高并发点赞”都成立。
- 但“Bloom 后优化为 Roaring Bitmap”当前仓库里没找到 `RoaringBitmap` 相关代码，所以不要讲成已经落地。
- 更稳妥的说法是：
  - “当前代码版本采用的是 Bloom + ZSET 方案，如果继续优化，Roaring Bitmap 是我考虑过的方向之一。”

## 九、用户关系服务

### 1. 业务上下文

用户关系服务负责维护社交关系链，主要是：

- 关注
- 取关
- 关注列表分页查询
- 粉丝列表分页查询

### 2. 关注链路

1. 先校验不能关注自己。
2. 调用户服务校验目标用户是否存在。
3. 构造当前用户关注列表的 Redis ZSET Key。
4. 执行 Lua 脚本 `follow_check_and_add.lua` 做原子校验和写入。
5. 如果 ZSET 不存在，就先从 DB 全量同步关注关系到 Redis，再补当前关注。
6. 最后通过 `asyncSendOrderly()` 发顺序 MQ，异步落库。

### 3. 取关链路

1. 先校验不能取关自己。
2. 校验目标用户存在。
3. 执行 `unfollow_check_and_delete.lua` 原子删除 Redis 中的关注关系。
4. 如果缓存不存在，就先从 DB 回填到 Redis，再做删除。
5. 最后发送顺序 MQ 异步落库。

### 4. 列表查询链路

#### 关注列表

- 优先查 Redis ZSET 的总量和分页数据。
- 命中后批量调用用户服务补全资料。
- 如果 Redis 没数据，则查 DB，并异步全量同步关注列表到 Redis。

#### 粉丝列表

- 逻辑类似，但会限制只同步和查询热点范围。
- 查出来后同时调用用户服务和计数服务补全用户信息及计数字段。

### 5. 为什么这样设计

- 关注/取关是高频写，先写 Redis、后异步落库，可以把写压力抗住。
- 用 Lua 脚本是为了把“是否已关注 + 更新 ZSET”做成原子操作，减少多次往返 Redis。
- 用顺序消息是为了尽量保证同一用户的关注/取关顺序一致，避免乱序导致最终状态错误。
- 关注列表和粉丝列表适合 ZSET，因为需要按时间倒序分页。

### 6. 你要能答出来的追问

#### 为什么关注关系不直接同步写 MySQL？

因为社交操作峰值很高，如果每次都直接写库，热点用户会把数据库打得很重。先写 Redis，后异步落库，更适合高并发。

#### 为什么 MQ 要按用户维度做顺序？

因为同一用户可能连续关注又取关同一个人，顺序消费更能保证最终状态不乱。

#### 为什么这里能说幂等？

因为 SQL 初始化脚本里有联合唯一索引：
- `t_following(user_id, following_user_id)`
- `t_fans(user_id, fans_user_id)`

所以即使消息重复消费，数据库层也有幂等约束。

### 7. 谨慎表述

- 这块“Redis + 顺序 MQ + Lua + 联合唯一索引幂等”可以放心讲。

## 十、计数服务

### 1. 业务上下文

关注数、粉丝数、点赞数、评论数、收藏数这些计数如果每次都直接改数据库，会非常重，所以项目把计数拆成独立服务。

### 2. 关键链路

以粉丝数为例：

1. 用户关系服务发关注/取关行为 MQ。
2. `CountFansConsumer` 消费消息后，不是立刻逐条落库，而是先进入 `BufferTrigger` 聚合。
3. 达到批量阈值或时间窗口后，按目标用户分组，算出每个用户最终应该加多少或减多少。
4. 先更新 Redis 里的计数 Hash。
5. 再发 MQ 到 `TOPIC_COUNT_FANS_2_DB`。
6. `CountFans2DBConsumer` 消费后先 `rateLimiter.acquire()` 限流，再批量更新数据库。

### 3. 为什么这样设计

- 先聚合再落库，可以把 1000 次小写合并成一次批处理。
- 先更新 Redis，用户查询时能更快看到近实时结果。
- 再限流写 DB，是典型的削峰填谷。

### 4. 你要能答出来的追问

#### BufferTrigger 是干什么的？

它是一个内存聚合器，可以按“条数”或“时间窗口”把消息攒成一批，再统一消费。这里就是用它把高频行为聚合成批量计数更新。

#### 为什么先更 Redis 再落库？

因为查询一般优先走 Redis，先更 Redis 能保证前台体验更接近实时；数据库则承担最终持久化。

#### RateLimiter 在这里的作用是什么？

它不是保证正确性，而是控制落库速率，避免消费过快把数据库打满。

### 5. 谨慎表述

- “BufferTrigger 1000 条一批、1 秒聚合一次、先 Redis 后 DB、RateLimiter 削峰”可以讲。
- 但要明确这是“最终一致性计数”，不是每一步都强一致。

## 十一、数据对齐服务

### 1. 业务上下文

既然计数服务走的是异步聚合，就意味着出现延迟、重复、漏更的可能性会增大，所以需要一个补偿系统做最终对齐。

### 2. 关键链路

1. 白天持续消费点赞、收藏、关注、发笔记等 MQ。
2. 只要某个对象当天发生过变更，就把它记录到“当日增量变更表”中。
3. 为了避免重复插入，先用 Lua + Bloom 过滤“今天这个对象是否已经记录过”。
4. 夜间由 XXL-Job 分片任务处理前一天的变更对象。
5. 对这些对象重新做真实 `count(*)` 计算。
6. 把结果回写正式计数表和 Redis。
7. 必要时重建 ES 文档，保证搜索侧也一致。

### 3. 为什么这样设计

- 不做全表重刷，因为成本太高。
- 只对“当天变过的对象”做重算，代价可控。
- 分片广播任务可以把大批量处理并行化，压缩夜间窗口。

### 4. 你要能答出来的追问

#### 为什么这里也用了 Bloom？

这里的 Bloom 不是为了精确判断业务状态，而是为了减少重复写入“当日变更记录”。最终准确值还是靠夜间重新 count。

#### 为什么需要预创建临时表？

因为对齐任务按天、按分片落表，提前建好表结构能让凌晨批处理更稳定，也减少运行时 DDL 风险。

### 5. 谨慎表述

- 你可以说“通过异步补偿实现最终一致性”。
- 不要说成“保证实时强一致”。

## 十二、搜索服务

### 1. 业务上下文

搜索服务负责两件事：

- 提供笔记搜索、用户搜索接口
- 维护 Elasticsearch 索引

### 2. 笔记搜索链路

1. 接口进入 `NoteSearchServiceImpl.searchNote()`。
2. 用 `multi_match` 搜标题和话题。
3. 可叠加笔记类型、发布时间范围等过滤条件。
4. 如果用户显式选择排序，则按发布时间、点赞数、评论数、收藏数排序。
5. 如果没指定排序，则走 `function_score` 自定义综合评分。
6. 返回结果时做标题高亮。

### 3. 用户搜索链路

1. 搜索昵称和小哈书号。
2. 默认按粉丝数降序。
3. 返回昵称高亮。

### 4. 增量索引链路

1. `CanalClient` 连接 Canal。
2. `CanalSchedule` 持续拉取 Binlog。
3. 按表名分流成用户事件和笔记事件。
4. 对插入、更新、删除分别做索引重建或删除。
5. 用户资料变化时，还会重建该用户相关的笔记索引，因为笔记文档里通常冗余了作者信息。

### 5. 为什么这样设计

- 搜索读请求不应该压 MySQL，所以核心查询走 ES。
- Binlog 增量同步比业务代码双写侵入更小，主业务只管落库，索引同步单独处理。
- 自定义评分适合内容平台，不只看文本相关度，还要看互动热度。

### 6. 你要能答出来的追问

#### 为什么标题权重更高？

因为标题通常比正文摘要更能表达用户意图，所以搜索相关性上应当给更高权重。

#### 为什么用户信息变更要重建笔记索引？

因为笔记索引里常常冗余作者昵称、头像等字段，用户资料一变，笔记文档也要更新。

### 7. 谨慎表述

- “Canal 增量索引、中文搜索、高亮、筛选、自定义评分”成立。
- “通过 logstash 实现全量索引构建”当前仓库没有看到对应配置，面试时不要当作已落地事实去讲。
- 更稳妥的说法是：
  - “当前代码里支持应用内重建索引，Canal 负责增量同步。”

## 十三、评论服务

### 1. 业务上下文

评论服务的特点是：

- 写多
- 查也多
- 一级评论和二级评论的读写模型不同
- 点赞、热度、评论数会互相联动

所以评论服务是这个项目里复杂度比较高的一块。

### 2. 评论发布链路

1. 校验正文和图片不能同时为空。
2. 调 ID 服务生成评论 ID。
3. 组装 `PublishCommentMqDTO`。
4. 使用 `SendMqRetryHelper.asyncSend()` 发 MQ。
5. `Comment2DBConsumer` 批量消费后，在事务里先写评论元数据，再批量写正文到 KV 服务。
6. 成功后同步一级评论热点 ZSET，并通知计数服务。

### 3. 评论查询链路

1. 先查 Redis 中的评论总数。
2. 一级评论列表优先查 Redis ZSET。
3. 评论详情优先查本地 Caffeine，再查 Redis，再查 DB。
4. 正文批量从 KV 服务查。
5. 二级评论分页也是类似链路，但缓存窗口更保守。

### 4. 点赞链路

1. 先校验评论存在。
2. 用 Bloom + Lua 判断是否可能点赞过。
3. 如果 Bloom 不存在，查 DB 初始化 Bloom。
4. 通过顺序 MQ 异步写点赞关系。
5. 消费端会把同一批次中同一用户的多次操作合并，只保留最终状态。

### 5. 为什么这样设计

- 评论发布接口快速返回，避免前台卡在数据库写入上。
- 评论查询读热点明显，适合本地缓存 + Redis + DB 的三级读路径。
- 评论点赞状态切换频繁，适合先 Redis 过滤，再 MQ 异步落库。
- 一级评论和二级评论读模型不同，所以缓存策略也不同。

### 6. 你要能答出来的追问

#### 为什么评论元数据和正文分离？

因为评论列表常查的是作者、时间、点赞数、热度、回复数这些结构化字段，而正文是大字段，拆开更利于读路径优化。

#### 为什么点赞消息还要“合并最后一次操作”？

因为用户可能在很短时间内点赞、取消点赞、再点赞，如果全量落库会有很多无效写。只保留最终态可以显著降低写放大。

#### 为什么删除评论还要广播删本地缓存？

因为 Caffeine 是每个节点本地的，Redis 删除不代表本地缓存自动失效。

### 7. 谨慎表述

- “异步发布、批量消费、批量写库、分页查询二级缓存、点赞异步批量落库”都成立。
- 但“生产者失败后的兜底写库机制”当前代码里是有设计意图、未完全落地，`fallback()` 还有 `TODO`。
- 所以面试时建议说：
  - “我实现了 Spring Retry + 异步重试的发送可靠性主链路，兜底持久化方案有设计，当前仓库版本没有完全补完。”
